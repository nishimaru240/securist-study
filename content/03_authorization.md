# 第3章: 認可・アクセス制御

この章では、認証済みユーザーに対して**何を許可するか**を管理する認可の仕組みを学ぶ。

- アクセス制御の基本と失敗例
- 限定公開URLのリスク
- OpenID Connect と OAuth の違い
- シングルサインオン（SSO）
- FIDO認証

---
# アクセス制御とは ★

**「誰が」「何を」「どう操作できるか」をシステムが判断・制御すること**

- 認証（誰か確認）の次のステップが認可（何を許可するか）
- アクセス制御が不備だと、**OWASP Top 10（2021）の A01：アクセス制御の不備**に該当

**代表的なアクセス制御モデル**

| モデル | 正式名称 | 意味 | 例 |
|--------|----------|------|----|
| DAC | Discretionary Access Control | 任意アクセス制御：オーナーが許可を設定 | UNIXファイルパーミッション |
| MAC | Mandatory Access Control | 強制アクセス制御：システムがラベルで制御 | 軍事システム |
| RBAC | Role-Based Access Control | ロールベース：役割に権限を割り当て | Webアプリの「管理者・一般ユーザー」 |

---
# アクセス制御の失敗例 ★

**よくある設計ミスと攻撃パターン**

【水平的権限昇格（IDOR：Insecure Direct Object Reference）】
- `/user/123/profile` を `/user/124/profile` に変えると他人の情報が見える
- 対策：URLのIDがセッションユーザーのものか**サーバー側で必ず検証する**

【垂直的権限昇格】
- 一般ユーザーが管理者ページに直接アクセスできてしまう
- 対策：すべてのリクエストで**権限チェックをサーバー側で実施**する（クライアント側のみの非表示は無効）

→ **UIで隠すだけはアクセス制御にならない。URLを直打ちされても拒否できる設計にする**

---
# 垂直的権限昇格の攻撃手口 ★

**「管理画面のメニューを非表示にしているだけ」では防げない**

- **URL直打ち**：`/admin/users` を推測してアドレスバーに直接入力
- **パラメータ改ざん**：リクエストの `role=user` を `role=admin` に書き換え、Cookie の権限フラグを改ざん
- **HTTPメソッド変更**：GETでは弾かれるが、`DELETE /api/users/123` には権限チェックが漏れている
- **強制ブラウジング（Forced Browsing）**：リンクされていない管理URL（`/admin`、`/manage`）を総当たりで発見

→ クライアントから送られた権限情報は**一切信頼しない**

---
# 権限昇格を防ぐ設計

**「すべてのリクエストでサーバー側が権限を判定する」が原則**

- 認可チェックを**共通ミドルウェア**にまとめて、個別実装漏れを防ぐ
- ホワイトリスト方式（許可された操作だけを通す）をデフォルトにする
- セッションから取得したユーザーIDとロールだけを信頼する（リクエストパラメータの role は無視）
- 管理機能は**専用エンドポイント**に分離し、ロール判定を入口で必ず実行する
- ログに権限拒否イベントを記録し、総当たり試行を検知できるようにする

# 限定公開URLの問題点

**限定公開URL（Capability URL）：URLを知っている人だけがアクセスできる設計**

例：`https://example.com/report/abc123xyz_private`

【問題点】
- URLがブラウザ履歴・**HTTPリファラー**（直前にいたページのURLを次のサイトへ自動送信するヘッダー）・サーバーログに残る
- URLを知っている人全員がアクセスできてしまう
- 誰かがシェアした瞬間に「限定」でなくなる

【正しい設計】
- URLランダム化だけに頼らず、**セッションベースの認証と組み合わせる**
- URLに有効期限を設け、一定期間後に無効化する

---
# OpenID Connect（OIDC）★

**OpenID Connect：OAuth 2.0 を拡張した「認証」プロトコル**

- ユーザーが「誰であるか」（認証）を第三者（IdP）に委任する仕組み
- 「Googleでログイン」「LINEでログイン」はOIDCを使っている
- 認証結果は **IDトークン（JWT形式：JSON Web Token、署名付きで改ざん検知できるトークン）** でアプリに渡される

**登場人物**

| 役割 | 名称 | 例 |
|------|------|----|
| ユーザー | End-User | ログインする人 |
| 認証サービス | IdP（Identity Provider） | Google、LINE、Entra ID |
| アプリ側 | RP（Relying Party） | あなたのWebアプリ |

---
# OpenID Connect の認証フロー

**Authorization Code フロー（最も安全、推奨）。2段階で別々のことを確認する**

【段階1：認可コード発行】＝「**ユーザー本人？このRPに権限を渡していい？**」

1. RPがIdPへリダイレクト（**state と nonce** を付与）
2. ユーザーがIdPでログイン・同意
3. IdPが**認可コード**（短命・1回限りの引換券）と state をブラウザ経由でRPに返す

【段階2：IDトークン発行】＝「**コードを持ってきたのは本当にそのRP？**」

4. RPが state を検証 → バックチャネルで認可コード＋**client_secret** をIdPへ送信
5. IdPがRP本人と確認 → **IDトークン（nonce 埋込）とアクセストークン**を発行
6. RPがIDトークンの**署名・有効期限・nonce・aud（受信者）を検証**してログイン完了

→ ブラウザ経由は「単独では無力な認可コード」だけ。IDトークン本体はサーバー間でしか流れない

---
# state と nonce の役割 ★

**双子の使い捨て乱数。発行タイミングは同じ（段階1のリダイレクト時）、配属先が違う**

| | state | nonce |
|---|---|---|
| 発行タイミング | IdPへのリダイレクトURL組み立て時 | 同左（同時に発行） |
| 載せる場所 | 認可リクエストURLのクエリ | 同左 |
| 戻ってくる場所 | 段階1：IdPからの**リダイレクトURL**に同梱 | 段階2：**IDトークンの中身**に埋込 |
| 検証タイミング | 認可コードを受け取った直後 | IDトークン受信・検証時 |
| 防ぐ攻撃 | **CSRF**（偽リダイレクトで他人の認可コードを掴まされる） | **トークン置換／リプレイ**（盗んだIDトークンの流用） |

- **CSRF**（Cross-Site Request Forgery）：被害者のログイン状態を悪用して意図しないリクエストを送らせる攻撃
- どちらもセッションに保管し、戻ってきた値と一致するかを検証する

---
# CSRF攻撃の手口（state がない場合）★

**「攻撃者が被害者になりすます」ではなく「被害者に攻撃者として操作させる」逆方向の攻撃**

1. 攻撃者が自分のアカウントでIdPにログイン・同意 → IdPから**コールバックURL**（認可コード付き）が生成される
2. 攻撃者はそのリダイレクトに**自分では従わず**、コールバックURLを被害者にクリックさせる
3. 被害者のブラウザがコールバックURLをRPに送信 → RPは被害者のセッションを**攻撃者のアカウントに紐付けてしまう**
4. 被害者は「自分のアカウントにログインした」と信じたまま操作＝**攻撃者のアカウントで操作している状態**
5. 被害者が入力したクレジットカード情報・住所等が攻撃者のアカウントに蓄積され、後から参照される

→ state があれば「自分が開始していない認証フローの応答」として弾ける

---
# トークン置換／リプレイ攻撃の手口（nonce がない場合）★

**流出した他人のIDトークンを、別の認証セッションに混入させる攻撃**

1. 攻撃者が有効期限内の他人のIDトークンを入手（ログ・キャッシュ流出、Implicitフローでの傍受など）
2. 攻撃者が自分でRPにログインを試み、IdPからの応答を**横取りした古いIDトークンに差し替え**てRPに渡す
3. RPは署名・有効期限はOKと判定 → **攻撃者を被害者として認証してしまう**

→ nonce があれば「自分が今回発行した値と違う＝今の認証要求に対応していない」として拒否できる
→ 特にIDトークンがフロントチャネル（URL）で流れる以下のフローで重要
- **Implicit フロー**：認可コードを経由せず、IdPがIDトークンをリダイレクトURLのフラグメント（`#`以降）に直接乗せて返す旧来の方式（現在は非推奨）
- **Hybrid フロー**：認可コードとIDトークンの両方をフロントチャネルで同時に返す方式

---
# OAuth 2.0 とは ★

**OAuth 2.0：「リソースへのアクセス権限」を第三者に委任するプロトコル**

- 認証ではなく**認可**が目的（「誰か」ではなく「何を許可するか」）
- 「Googleカレンダーへのアクセスを許可する」などがOAuthの典型例

**OAuth 2.0 の登場人物**

| 役割 | 意味 |
|------|------|
| Resource Owner | リソースの所有者（=ユーザー） |
| Client | 権限委任を受けたいアプリ |
| Authorization Server | アクセストークンを発行するサーバー |
| Resource Server | アクセストークンで保護されたAPIサーバー |

**OIDCとOAuthの違い（混同しやすい）**

| 目的 | プロトコル | 発行されるもの |
|------|------------|----------------|
| 認証（誰か確認） | OpenID Connect | IDトークン（JWT） |
| 認可（権限委任） | OAuth 2.0 | アクセストークン |

---
# OAuth 2.0 の主なグラントタイプ

**グラントタイプ：アクセストークンを取得する方法の種類**

| タイプ | 用途 | 推奨 |
|--------|------|------|
| Authorization Code | Webアプリ（サーバーサイド） | ✅ |
| Authorization Code + PKCE | SPAやモバイルアプリ | ✅ |
| Client Credentials | サーバー間通信（M2M） | ✅ |
| Implicit | SPAの旧来設計 | ❌ 非推奨（トークン漏洩リスク） |
| Resource Owner Password | 直接パスワードを渡す | ❌ 非推奨 |

- **PKCE（Proof Key for Code Exchange）**：クライアントが認証要求時にランダムな秘密値を生成し、その秘密値を知っている本人だけが認可コードをトークンに交換できる仕組み。SPA・モバイルアプリは秘密鍵を安全に保管できないため、認可コードを横取りされてもトークン取得を防げるPKCEが必須

→ 新規設計では **Authorization Code + PKCE** を使う

---
# シングルサインオン（SSO）

**SSO（Single Sign-On）：一度の認証で複数のシステムにアクセスできる仕組み**

- 社内ポータル・勤怠・経費精算などを個別にログインしなくて済む
- ユーザビリティ向上とパスワード管理の負荷軽減が目的

**主なSSOの実装方式**

| 方式 | 特徴 |
|------|------|
| SAML 2.0 | 企業向け。XMLベース。レガシーに多い |
| OpenID Connect | モダン。JSONベース。現在の主流 |
| Kerberos | オンプレのAD環境でよく使われる |

---
# SAML と Kerberos の概要

- **SAML 2.0**（Security Assertion Markup Language）：認証情報をXMLで表現してIdPとRP間でやり取りする規格。古くから企業が採用してきたためレガシーシステムに多く残っている
- **Kerberos**：社内ネットワーク（Active Directory）で使われる認証プロトコル。鍵を持った認証サーバーが「通行証（チケット）」を発行し、各サーバーはそれを見て本人と判断する仕組み
- **OpenID Connect**：第3章前半で扱った通り、JSONベースで現代のWebサービスに適した認証規格。新規設計では基本これを選ぶ

---
# SSO のセキュリティ設計

**SSOは「1点突破で全システム」へのリスクがある**

【主なリスク】
- IdPが侵害されると、すべての連携サービスが被害を受ける
- セッションタイムアウトの管理が複雑になる

【対策設計】
- 重要操作（送金・設定変更等）には**ステップアップ認証**（再認証・追加要素）を入れる
- IdP側でも多要素認証（MFA）を強制する
- シングルログアウト（SLO）を実装して全連携先のセッションを一括無効化する

# FIDO認証 ★

**FIDO（Fast IDentity Online：パスワード不要の認証標準）**

- パスワードを使わず、**公開鍵暗号方式**で認証する
- デバイス上の秘密鍵がサーバーに渡らないため、フィッシングが成立しない

**FIDO2の構成要素**

| 要素 | 役割 |
|------|------|
| WebAuthn | Webブラウザとサーバー間の認証API仕様 |
| CTAP | デバイス（生体認証器・セキュリティキー）との通信仕様 |

---
# FIDO認証の仕組みと利点

**登録フロー：サーバーに公開鍵だけを預ける**

1. ユーザーがデバイス（スマホ・セキュリティキー）で鍵ペアを生成
2. 公開鍵をサーバーに登録（秘密鍵はデバイスから出ない）
3. 認証時：サーバーからのチャレンジに秘密鍵で署名 → 公開鍵で検証

**利点**
- パスワード漏洩リスクがゼロ（そもそも存在しない）
- フィッシングサイトに誘導されても秘密鍵は渡らない
- 生体認証（顔・指紋）とセットで使いやすい

---
# 第3章 まとめ

- アクセス制御はサーバー側で必ず検証（UIで隠すだけは無効）
- 水平的・垂直的権限昇格のリスクをパラメータ検証で防ぐ
- 限定公開URLはセッション認証と組み合わせて有効期限を設ける
- **OpenID Connect**：認証の委任（IDトークンを受け取る）
- **OAuth 2.0**：認可の委任（アクセストークンを受け取る）
- **FIDO2（WebAuthn）**：パスワード不要・フィッシング耐性の高い認証
