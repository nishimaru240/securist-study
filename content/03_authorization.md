# 第3章: 認可・アクセス制御

この章では、認証済みユーザーに対して**何を許可するか**を管理する認可の仕組みを学ぶ。

- アクセス制御の基本と失敗例
- 限定公開URLのリスク
- OpenID Connect と OAuth の違い
- シングルサインオン（SSO）
- FIDO認証

---
# アクセス制御とは ★

**「誰が」「何を」「どう操作できるか」をシステムが判断・制御すること**

- 認証（誰か確認）の次のステップが認可（何を許可するか）
- アクセス制御が不備だと、**OWASP Top 10（2021）の A01：アクセス制御の不備**に該当

**代表的なアクセス制御モデル**

| モデル | 意味 | 例 |
|--------|------|----|
| DAC | 任意アクセス制御：オーナーが許可を設定 | UNIXファイルパーミッション |
| MAC | 強制アクセス制御：システムがラベルで制御 | 軍事システム |
| RBAC | ロールベース：役割に権限を割り当て | Webアプリの「管理者・一般ユーザー」 |

---
# アクセス制御の失敗例 ★

**よくある設計ミスと攻撃パターン**

【水平的権限昇格（IDOR：Insecure Direct Object Reference）】
- `/user/123/profile` を `/user/124/profile` に変えると他人の情報が見える
- 対策：URLのIDがセッションユーザーのものか**サーバー側で必ず検証する**

【垂直的権限昇格】
- 一般ユーザーが管理者ページに直接アクセスできてしまう
- 対策：すべてのリクエストで**権限チェックをサーバー側で実施**する（クライアント側のみの非表示は無効）

→ **UIで隠すだけはアクセス制御にならない。URLを直打ちされても拒否できる設計にする**

---
# 限定公開URLの問題点

**限定公開URL（Capability URL）：URLを知っている人だけがアクセスできる設計**

例：`https://example.com/report/abc123xyz_private`

【問題点】
- URLがブラウザ履歴・HTTPリファラー・サーバーログに残る
- URLを知っている人全員がアクセスできてしまう
- 誰かがシェアした瞬間に「限定」でなくなる

【正しい設計】
- URLランダム化だけに頼らず、**セッションベースの認証と組み合わせる**
- URLに有効期限を設け、一定期間後に無効化する

# OpenID Connect（OIDC）★

**OpenID Connect：OAuth 2.0 を拡張した「認証」プロトコル**

- ユーザーが「誰であるか」（認証）を第三者（IdP）に委任する仕組み
- 「Googleでログイン」「LINEでログイン」はOIDCを使っている
- 認証結果は **IDトークン（JWT形式）** でアプリに渡される

**登場人物**

| 役割 | 名称 | 例 |
|------|------|----|
| ユーザー | End-User | ログインする人 |
| 認証サービス | IdP（Identity Provider） | Google、LINE、Entra ID |
| アプリ側 | RP（Relying Party） | あなたのWebアプリ |

---
# OpenID Connect の認証フロー

**Authorization Code フロー（最も安全、推奨）**

1. ユーザーがRPにアクセス → RPがIdPにリダイレクト（**state パラメータ**を付与）
2. ユーザーがIdP上でログイン・同意
3. IdPがRPに「認可コード」をリダイレクトで返す（state を含む）
4. RPは state を検証してから、バックチャンネル（サーバー間）で認可コードをトークンに交換
5. RPがIDトークンの**署名・有効期限・nonce・aud（受信者）を検証**してログイン完了

→ **トークンをブラウザ経由で渡さない**ためトークン漏洩リスクが低い  
→ **state はCSRF対策**、**nonce はトークン置換攻撃対策**として必須

---
# OAuth 2.0 とは ★

**OAuth 2.0：「リソースへのアクセス権限」を第三者に委任するプロトコル**

- 認証ではなく**認可**が目的（「誰か」ではなく「何を許可するか」）
- 「Googleカレンダーへのアクセスを許可する」などがOAuthの典型例

**OAuth 2.0 の登場人物**

| 役割 | 意味 |
|------|------|
| Resource Owner | リソースの所有者（=ユーザー） |
| Client | 権限委任を受けたいアプリ |
| Authorization Server | アクセストークンを発行するサーバー |
| Resource Server | アクセストークンで保護されたAPIサーバー |

**OIDCとOAuthの違い（混同しやすい）**

| 目的 | プロトコル | 発行されるもの |
|------|------------|----------------|
| 認証（誰か確認） | OpenID Connect | IDトークン（JWT） |
| 認可（権限委任） | OAuth 2.0 | アクセストークン |

---
# OAuth 2.0 の主なグラントタイプ

**グラントタイプ：アクセストークンを取得する方法の種類**

| タイプ | 用途 | 推奨 |
|--------|------|------|
| Authorization Code | Webアプリ（サーバーサイド） | ✅ |
| Authorization Code + PKCE | SPAやモバイルアプリ | ✅ |
| Client Credentials | サーバー間通信（M2M） | ✅ |
| Implicit | SPAの旧来設計 | ❌ 非推奨（トークン漏洩リスク） |
| Resource Owner Password | 直接パスワードを渡す | ❌ 非推奨 |

→ 新規設計では **Authorization Code + PKCE** を使う

# シングルサインオン（SSO）

**SSO（Single Sign-On）：一度の認証で複数のシステムにアクセスできる仕組み**

- 社内ポータル・勤怠・経費精算などを個別にログインしなくて済む
- ユーザビリティ向上とパスワード管理の負荷軽減が目的

**主なSSOの実装方式**

| 方式 | 特徴 |
|------|------|
| SAML 2.0 | 企業向け。XMLベース。レガシーに多い |
| OpenID Connect | モダン。JSONベース。現在の主流 |
| Kerberos | オンプレのAD環境でよく使われる |

---
# SSO のセキュリティ設計

**SSOは「1点突破で全システム」へのリスクがある**

【主なリスク】
- IdPが侵害されると、すべての連携サービスが被害を受ける
- セッションタイムアウトの管理が複雑になる

【対策設計】
- 重要操作（送金・設定変更等）には**ステップアップ認証**（再認証・追加要素）を入れる
- IdP側でも多要素認証（MFA）を強制する
- シングルログアウト（SLO）を実装して全連携先のセッションを一括無効化する

# FIDO認証 ★

**FIDO（Fast IDentity Online：パスワード不要の認証標準）**

- パスワードを使わず、**公開鍵暗号方式**で認証する
- デバイス上の秘密鍵がサーバーに渡らないため、フィッシングが成立しない

**FIDO2の構成要素**

| 要素 | 役割 |
|------|------|
| WebAuthn | Webブラウザとサーバー間の認証API仕様 |
| CTAP | デバイス（生体認証器・セキュリティキー）との通信仕様 |

---
# FIDO認証の仕組みと利点

**登録フロー：サーバーに公開鍵だけを預ける**

1. ユーザーがデバイス（スマホ・セキュリティキー）で鍵ペアを生成
2. 公開鍵をサーバーに登録（秘密鍵はデバイスから出ない）
3. 認証時：サーバーからのチャレンジに秘密鍵で署名 → 公開鍵で検証

**利点**
- パスワード漏洩リスクがゼロ（そもそも存在しない）
- フィッシングサイトに誘導されても秘密鍵は渡らない
- 生体認証（顔・指紋）とセットで使いやすい

---
# 第3章 まとめ

- アクセス制御はサーバー側で必ず検証（UIで隠すだけは無効）
- 水平的・垂直的権限昇格のリスクをパラメータ検証で防ぐ
- 限定公開URLはセッション認証と組み合わせて有効期限を設ける
- **OpenID Connect**：認証の委任（IDトークンを受け取る）
- **OAuth 2.0**：認可の委任（アクセストークンを受け取る）
- **FIDO2（WebAuthn）**：パスワード不要・フィッシング耐性の高い認証
