# 第6章: 出力処理

この章では、アプリケーションからブラウザやDBへの**出力**を安全に行う設計を学ぶ。

- XSS対策（HTMLエスケープ）
- SQLインジェクション対策
- JSON出力の注意点
- OSコマンドインジェクション対策
- HTTPレスポンスヘッダーの管理

---
# 出力処理の基本原則

**出力先に応じて適切なエスケープを適用する**

- 入力値の検証だけでは不十分 → **出力する文脈（Context）に合わせた変換が必要**

| 出力先 | エスケープ種別 |
|--------|----------------|
| HTMLの本文・属性 | HTMLエスケープ |
| SQLクエリ | プリペアドステートメント |
| JavaScriptコード | JSエスケープ |
| URLパラメーター | URLエンコード |
| OSコマンド | コマンド呼び出し自体を避ける |

---
# XSS（クロスサイトスクリプティング）★

**XSS：Webページに悪意あるスクリプトを埋め込み、閲覧者のブラウザで実行させる攻撃**

【主なXSSの種類】

| 種類 | 特徴 |
|------|------|
| 反射型（Reflected） | URLに含まれたスクリプトをそのまま出力 |
| 蓄積型（Stored） | DBに保存したスクリプトを後から出力 |
| DOM型 | クライアント側のJSがDOMを書き換えて発生 |

【被害】Cookieの窃取・フィッシングページへのリダイレクト・キーロガー埋め込み

> **DOM**（Document Object Model）：ブラウザがHTMLを解析して作る木構造のデータ。JavaScriptはDOMを操作することでページの内容を動的に書き換えられる。DOM型XSSはサーバーを経由せずブラウザ内だけで完結するため、サーバー側のエスケープが効かない
> **キーロガー**：ユーザーのキーボード入力を記録するスクリプト。XSSで埋め込まれるとパスワード・クレジットカード番号等の入力内容がそのまま攻撃者に送信される

---
# XSS対策：HTMLエスケープの基本 ★

**エスケープ：HTMLの特殊文字を無害な文字列に置換してブラウザに文字として表示させる処理**

出力が必要なケース：コメント欄・検索結果・プロフィール等、ユーザー入力をページに表示する箇所すべて

| 文字 | エスケープ後 | 理由 |
|------|--------------|------|
| `<` | `&lt;` | タグの開始と誤認される |
| `>` | `&gt;` | タグの終了と誤認される |
| `&` | `&amp;` | エンティティ記法の開始と誤認される |
| `"` | `&quot;` | 属性値の終端と誤認される |
| `'` | `&#x27;` | 属性値の終端と誤認される |

---
# XSS対策：安全な実装方法

**テンプレートエンジンを使う（サーバー側）**
- Jinja2・Thymeleaf等の自動エスケープ機能を有効にする → 手動エスケープ漏れを防ぐ

**JavaScriptでの出力方法（クライアント側）**

| 方法 | 動作 | XSSリスク |
|------|------|-----------|
| `innerHTML` | 文字列をHTMLとして解析・描画 | あり（`<script>` が実行される）❌ |
| `textContent` | 文字列をテキストとしてそのまま表示 | なし（タグも文字列として表示）✅ 推奨 |

**HTMLの装飾を許可したい場合（コメント欄でリンクや太字を使わせる等）**
- HTMLサニタイザーライブラリで危険なタグのみ除去する（例：DOMPurify・Bleach）
- 自前で `<script>` を除去しようとするとバイパスされやすいため、ライブラリに任せる

---
# 出力文脈ごとのエスケープ

**HTMLエスケープだけでは不十分。「どこに出力するか」で必要な処理が違う**

| 出力文脈 | エスケープ方法 |
|----------|----------------|
| HTML本文 | `<` `>` `&` 等をHTMLエンティティ化 |
| HTML属性 | `"` `'` も含めてエンティティ化、属性は必ずダブルクォートで囲む |
| URL（href・src） | URLエンコード＋`http://` `https://` 限定（`javascript:` 拒否） |
| JavaScript文字列 | `\u00XX` 形式でユニコードエスケープ |
| CSS値 | 16進数エスケープ |

---
# 出力文脈ごとのエスケープ：HTML属性

**HTML属性にユーザー入力を出力する場合**

```html
<input value="ユーザー入力">
```

攻撃者が `" onmouseover="alert(1)` と入力すると：

```html
<input value="" onmouseover="alert(1)">
```

→ `"` で属性値を閉じてイベントハンドラを注入できる。`"` `'` もエスケープし、属性を必ずクォートで囲むことで防ぐ

---
# 出力文脈ごとのエスケープ：URL

**URLにユーザー入力を出力する場合**

```html
<a href="ユーザー入力">リンク</a>
```

攻撃者が `javascript:alert(document.cookie)` と入力すると：

```html
<a href="javascript:alert(document.cookie)">リンク</a>
```

→ クリックした瞬間にJSが実行される。`http://` `https://` 以外のスキームを拒否することで防ぐ

---
# 出力文脈ごとのエスケープ：JS・CSS

**JavaScript文字列にユーザー入力を出力する場合**

```html
<script>var name = "ユーザー入力";</script>
```

攻撃者が `"; alert(1); //` と入力すると文字列が閉じられてスクリプトが挿入される。`\u0022`（ダブルクォートのユニコードエスケープ）に変換することで防ぐ

**CSS値にユーザー入力を出力する場合**

CSSには `url()` や `expression()` があり、スクリプト実行に悪用される。16進数エスケープで無害化する

> **16進数エスケープ**：`;` や `}` など特殊文字を `\3b` `\7d` のような16進数表記に変換する処理。変換後はブラウザが文字として描画するだけでCSSの構文として解釈されない。現代ブラウザではCSSからのスクリプト実行はほぼブロックされているが、文脈固有エスケープの原則として押さえておく

---
# Content Security Policy（CSP）

**CSP（コンテンツセキュリティポリシー）：ブラウザが読み込めるリソースをサーバーが制限する仕組み**

- XSSが成功してもスクリプトの実行をブロックできる（多層防御）
- `Content-Security-Policy` レスポンスヘッダーで指定する

**設定例**
```
Content-Security-Policy: default-src 'self'; script-src 'self'
```
- `'self'` = 同一オリジン（同じプロトコル・ドメイン・ポートの組み合わせ）のリソースのみ許可
- インラインスクリプト（外部 `.js` ファイルではなく、HTML内に直接書いた `<script>…</script>` や `onclick="..."` などのスクリプト）の実行を禁止できる

# SQLインジェクション ★

**SQLインジェクション：ユーザー入力をSQL文に埋め込むことで、不正なSQLを実行させる攻撃**

【脆弱なコード例】
```python
sql = "SELECT * FROM users WHERE name='" + name + "'"
```
→ `name` に `' OR '1'='1` を入れると実際のSQLは以下になる：
```sql
SELECT * FROM users WHERE name='' OR '1'='1'
```
`'1'='1'` は常にTRUEのためWHERE句全体がTRUEになり、全件取得される

【被害】
- 全データの流出・改ざん・削除
- 管理者アカウントの乗っ取り
- サーバーへの踏み台

---
# SQLインジェクション対策 ★

**プリペアドステートメント（パラメータ化クエリ）を使う**

```python
# 安全なコード
cursor.execute("SELECT * FROM users WHERE name = %s", (name,))
```

- パラメーターとSQL構造を分離して送るため、入力値がSQL構文として解釈されない
- **フレームワークのORM**（Object-Relational Mapping：DBのテーブルをプログラムのオブジェクトとして扱える仕組み）**を使えば自動的にパラメータ化される**（ただし Django ORM の `raw()`・SQLAlchemy の `text()` など生SQL直書きAPIはパラメータ化されないため注意）

**追加の対策**
- DBアカウントは最小権限にする（アプリ用アカウントにDROP権限を与えない）
- エラーメッセージにSQLの詳細を表示しない

---
# JSON出力の注意点

**JSONをHTMLに埋め込む場合も文字エスケープが必要**

- JSONの `<`, `>`, `&` がHTML文脈でHTMLタグとして解釈される可能性がある
- JSONを `<script>` タグ内に埋め込む設計は避ける

**安全な設計**
- APIレスポンスは `Content-Type: application/json` を明示する
- `<`, `>` をUnicodeエスケープして出力する
- JSONP（クロスドメインでJSONを取得する古い手法。`<script>`タグでJSONをJavaScriptとして読み込む）は使わない（CORS対応のAPIに切り替える）

---
# OSコマンドインジェクション ★

**OSコマンドインジェクション：ユーザー入力をOSコマンドに渡すことで任意コマンドを実行させる攻撃**

【脆弱なコード例】
```python
os.system("ping " + user_input)
```
→ `user_input` に `; rm -rf /` を入れると任意コマンドが実行される

【対策（優先順）】
1. **OSコマンドの呼び出し自体をやめる**（言語標準ライブラリで代替）
2. シェル経由の実行を避ける（`subprocess.run(["ping", ip])` のようにリスト渡し）
3. 入力値をホワイトリストで検証してから渡す

---
# HTTPヘッダーインジェクション（CRLFインジェクション）

**HTTPヘッダーインジェクション（CRLFインジェクション）：レスポンスヘッダーに改行（CR・LF）を埋め込む攻撃**

- ヘッダーの区切りはCRLF（`\r\n`） → 入力に改行を含めると**ヘッダーやボディを偽造**できる
- リダイレクト先URLや Set-Cookie の値にユーザー入力をそのまま使うと発生する
- **HTTPレスポンス分割**（HTTP Response Splitting）：CRLFインジェクションを使って1つのHTTPレスポンスを2つに分割し、2つ目を完全に攻撃者が制御できる状態にする発展的な攻撃（キャッシュポイズニングや偽コンテンツ配信に悪用される）

【被害】
- 任意のCookieをセットされる（セッション固定攻撃の踏み台）
- 偽のレスポンスを表示させる（キャッシュポイズニング）

【対策】
- ユーザー入力を**ヘッダーに直接出さない**
- フレームワークのレスポンスAPI（`Location` 設定関数等）を使う（APIの内部でCR・LFを自動的に除去・拒否する処理が組み込まれているため、手動でヘッダー文字列を組み立てるより安全）
- 入力に CR・LF が含まれていたら拒否する

---
# HTTPレスポンスヘッダーの管理 ★

**セキュリティ関連の主要レスポンスヘッダー**

| ヘッダー | 目的 |
|----------|------|
| `Content-Security-Policy` | XSSの実行をブロック |
| `X-Content-Type-Options: nosniff` | MIMEタイプのスニッフィング防止 |
| `X-Frame-Options: DENY` | クリックジャッキング対策（第8章でも扱う） |
| `Strict-Transport-Security` | HTTPS強制（HSTS） |
| `Cache-Control: no-store` | 機密情報のキャッシュを禁止 |

> **MIME**（Multipurpose Internet Mail Extensions）：ファイルの種類を示す規格（例：`text/html`、`application/javascript`）。**MIMEスニッフィング**：ブラウザが `Content-Type` ヘッダーを無視してファイルの中身を読み種類を推測する動作。`nosniff` を指定するとこの推測を禁止し、宣言した `Content-Type` 通りに処理させる（スクリプトとして実行される誤作動を防ぐ）

- `Server` ヘッダーはバージョン情報を削除する（情報漏洩対策）

# オープンリダイレクト ★

**オープンリダイレクト：ユーザー指定URLにそのままリダイレクトする実装の脆弱性**

【攻撃例】
```
https://example.com/redirect?to=https://evil.com/fake-login
```
- 正規ドメインで始まるURL → ユーザーは安心してクリック
- リダイレクト先で本物そっくりのログイン画面 → 認証情報を窃取（**フィッシングの踏み台**）

【被害】
- 自社ドメインの信頼性を利用したフィッシング詐欺
- 認可コード・トークンをリダイレクトで盗まれる（OAuth/OIDC の認証フロー悪用）

---
# オープンリダイレクト対策

- リダイレクト先を**ホワイトリスト**で限定する（許可ドメイン or 内部パスのみ）
- 外部URLを受け付ける場合は**スキームを `http://` `https://` に限定**し、`javascript:` を拒否
- リダイレクト先IDで内部マッピング（`?to=mypage` → サーバー側で実URLを解決）
- 外部サイトへ遷移する場合は**ワンクッション**（「外部サイトへ移動します」確認ページ）を挟む
- フレームワークのリダイレクトAPIに**相対パスのみ**渡す設計にする

→ OIDC/OAuth の `redirect_uri` も同じ理由で**完全一致のホワイトリスト**にする

# 第6章 まとめ ★

- 出力先に応じたエスケープを使う（HTML・SQL・URL・JSで方法が違う）
- **HTMLエスケープ**でXSSを防ぐ。テンプレートエンジンの自動エスケープを活用
- **プリペアドステートメント**でSQLインジェクションを防ぐ
- **OSコマンド呼び出し自体を避ける**設計が最善
- ヘッダーへの入力出力時はCR・LFを拒否してHTTPヘッダーインジェクションを防ぐ
- リダイレクト先は**ホワイトリスト**で制限してオープンリダイレクトを防ぐ
- CSPヘッダーを設定してXSSの多層防御を行う
- セキュリティ関連レスポンスヘッダーを網羅的に設定する
