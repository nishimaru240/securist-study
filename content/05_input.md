# 第5章: 入力処理

この章では、ユーザーやシステムからの入力データを**安全に処理する**ための設計を学ぶ。

- クライアント検証とサーバー検証の役割の違い
- 文字種・文字長の検証
- 文字エンコーディングの統一
- ファイルアップロード・XML・デシリアライズの危険と対策

---
# 入力検証の基本原則 ★

**入力値は「信頼できない」前提で扱う**

- ユーザーが送信するデータはすべて不正入力の可能性がある
- ネットワーク越しに来るデータは改ざんされている可能性もある

**検証の場所**

| 場所 | 役割 | 重要度 |
|------|------|--------|
| クライアント側（JS） | UX向上（即時フィードバック） | 補助的 |
| サーバー側 | セキュリティ保証 | **必須** |

→ **クライアント側の検証はバイパスできる。サーバー側で必ず再検証する**

---
# クライアント側検証がバイパスされる理由

**攻撃者はブラウザのJavaScriptを迂回してリクエストを送れる**

【迂回手段】
- ブラウザの開発者ツールでJSを無効化
- curlやBurp Suiteなどのツールで直接リクエストを送信
- JavaScriptを書き換えてバリデーションをスキップ

【設計上のルール】
- クライアント検証はUX目的と割り切る
- セキュリティ上の判断は**すべてサーバー側で行う**
- 「hidden フィールドの値は変更されない」という前提は禁物

---
# 文字種・文字長の検証 ★

**入力値は「許可する文字・長さ」を明示的に定義する（ホワイトリスト方式）**

| 項目 | 検証内容 | 例 |
|------|----------|----|
| 文字種 | 使用できる文字の種類 | 半角英数字のみ、日本語不可 |
| 文字長 | 最小・最大長 | 8〜128文字 |
| フォーマット | 入力値の形式 | メール形式、電話番号形式 |
| 数値範囲 | 数値の最小・最大値 | 0〜9999の整数 |

- 「禁止文字を除去する（ブラックリスト）」より「許可文字のみ受け付ける（ホワイトリスト）」が安全
- 最大長を設定しないとバッファオーバーフロー（用意した記憶領域を超える長いデータを書き込まれて隣の領域を壊される攻撃）やDoS攻撃の踏み台になる

---
# 文字エンコーディングの統一

**文字エンコーディング：文字をバイト列に変換する方式**

- 異なるエンコーディングが混在すると、文字化けや**セキュリティホール**が生まれる
- 例：Shift-JISの一部のバイト列がUTF-8の特殊文字と衝突してエスケープが無効化される

**統一設計**

- アプリ全体で **UTF-8** に統一する
- DBの文字コードもUTF-8（MySQL: `utf8mb4`）に統一する
- 入力受付時にエンコーディングを確認・変換する処理を入れる
- `Content-Type: text/html; charset=UTF-8` を必ずレスポンスに付ける

---
# 数値・整数の検証

**数値入力には型・範囲・桁数の検証を行う**

【ありがちな問題】
- 整数オーバーフロー：許容値を超えた数値を入れて計算結果が異常になる
- 負の数の悪用：商品個数に `-1` を入れて金額を不正に減らす
- 巨大な数値：`9999999999...` を渡して処理を止める（DoS）

【対策】
- 整数型・小数型・符号あり/なしを明示してパースする
- 業務上の最小値・最大値で範囲チェックする
- 入力値が想定の桁数を超えたら拒否する

---
# ファイルアップロードの危険 ★

**アップロード機能は複数の攻撃手段になる**

【危険なファイルの種類】
- **サーバーサイドスクリプト**（PHP等）：アップロード後にURLでアクセスされて任意コード実行
- **HTMLファイル**：開くとブラウザでJavaScriptが動きXSSの踏み台になる
- **巨大ファイル**：ディスク容量を使い切らせるDoS攻撃
- **偽装ファイル**：拡張子だけ画像にして中身は実行ファイル

→ 拡張子チェックだけでは防げない（ファイル名はユーザーが自由に変えられる）

---
# ファイルアップロードの対策 ★

- **マジックナンバー検証**：ファイルの先頭数バイトを読んで実際の種類を判定する（PNGは`89 50 4E 47`で始まる等）。ブラウザが送る `Content-Type` は信用しない
- **公開ディレクトリの外に保存**：Webサーバーが直接URLで返せない場所に置き、ダウンロードはアプリ経由にする → アップロードしたPHPファイルが実行されるのを防ぐ
- **ファイル名をサーバー側で付け直す**：ユーザー指定の名前は使わない（パストラバーサル・上書き対策）
- **サイズ上限を設定**：DoS対策
- **画像は再エンコード**：サーバー側でリサイズ・変換し、埋め込みスクリプトを除去する

---
# パス・トラバーサル攻撃

**パス・トラバーサル：`../` を使ってサーバーの意図しないファイルにアクセスする攻撃**

【攻撃例】
```
GET /file?name=../../etc/passwd
```
→ `/var/www/uploads/../../etc/passwd` = `/etc/passwd` を読み取られる

【対策】
- ファイル名に `../` や絶対パスが含まれないか検証する
- ファイルパスをアクセス許可ディレクトリと比較して範囲外を拒否する（パス正規化後に検証）
- ユーザー入力をファイルパスに直接使わず、**IDでマッピング**する設計にする

---
# XMLの仕組み：エンティティとDTD

**XMLで使われる2つの概念**

- **エンティティ**：XMLの中で使える「変数」のようなもの。`&xxe;` のように書くと定義した値に置き換わる
- **DTD（Document Type Definition）**：XMLの構造やエンティティを定義するブロック。`<!DOCTYPE ...>` で始まる
- **外部エンティティ**：エンティティの参照先を外部ファイルやURLに指定できる機能。`SYSTEM "file:///etc/passwd"` のように書くとサーバー上のファイルを参照できる

→ 外部エンティティは便利な機能だが、**悪用するとサーバー内のファイルを読み出せる**

---
# XXE攻撃 ★

**XXE（XML External Entity）：外部エンティティ参照を悪用したサーバー内ファイルの読み取り攻撃**

【攻撃例（悪意あるXML）】
```xml
<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<root>&xxe;</root>
```
→ サーバーが `&xxe;` を展開する際に `/etc/passwd` を読み込み、内容をレスポンスとして返す

---
# XXE攻撃への対策

【対策】
- **DTD処理ごと無効化する**：XMLパーサーの設定で `<!DOCTYPE>` 自体を処理しないようにする（外部エンティティだけでなくDTD全体を無効にするのが確実）
- 可能であればXMLではなくJSONを使用する
- 信頼できないソースからのXMLは受け付けない

---
# SSRF（サーバーサイドリクエストフォージェリ）★

**SSRF：サーバーに任意の宛先へリクエストを送らせる攻撃（OWASP Top 10 A10）**

【攻撃の例】
- ユーザーが URL を入力できるフォーム（画像取得・Webhook等）に、社内のIPアドレスを指定
  - 例：`http://169.254.169.254/...`（クラウドのメタデータエンドポイント）
- 外部からアクセスできない**社内システム・クラウドの認証情報**を窃取される

【対策】
- ユーザーが指定したURLへリクエストする前にホスト名を解決し、**プライベートIP・ループバックを拒否**する（プライベートIP＝`192.168.x.x` など社内ネットワーク専用のアドレス、ループバック＝`127.0.0.1` などサーバー自身を指すアドレス）
- ホワイトリスト方式で**許可ドメインのみ**に通信先を絞る
- HTTPリダイレクトの追跡を無効化する（リダイレクト先が内部に向くのを防ぐ）

---
# デシリアライズ攻撃と対策

**デシリアライズ（Deserialization）：バイト列やJSON等のデータをオブジェクトに復元する処理**

**安全でないデシリアライズ（OWASP A08）：信頼できないデータをそのままデシリアライズすると任意コードが実行される**

【攻撃の仕組み】
- 攻撃者が細工したオブジェクトをシリアライズ（オブジェクトを保存・送信できるバイト列に変換すること）してサーバーに送信
- デシリアライズ時にマジックメソッド（`__wakeup__` 等）が呼ばれてコードが実行される

---
# デシリアライズ攻撃の成立条件

攻撃が成立するには以下の2条件が同時に揃う必要がある。

1. **マジックメソッドを持つクラスがサーバー上に存在する**（開発者が書いた正規のコード）
2. **信頼できないデータをそのままデシリアライズしている**（入力検証なし）

攻撃者は「そのクラス＋任意のプロパティ値」を組み立てたシリアライズデータを送るだけで、サーバー上の既存コードを好きなパラメーターで実行させられる。

**対策との対応**
- 「デシリアライズしない」→ 条件2を潰す
- 「HMACで検証する」→ 自分が送ったデータかを確認して条件2を潰す

【対策】
- 信頼できないソースからのシリアライズデータを**デシリアライズしない**
- デシリアライズ前にHMAC（共通鍵で計算する改ざん検出用の署名）等で「データが自分が送ったものか」を検証する
- Java の `ObjectInputStream` など危険なクラスの使用を避ける

---
# 第5章 まとめ ★

- 入力検証はサーバー側で必ず実施（クライアント側のみはバイパス可能）
- 検証は**ホワイトリスト方式**（許可する文字・長さ・形式を明示）
- 文字エンコーディングはアプリ・DB全体で **UTF-8 に統一**する
- ファイルアップロードはマジックナンバー検証＋公開ディレクトリ外への保存（MIMEタイプはユーザーが書き換え可能なため信頼しない）
- XMLは**外部エンティティ参照を無効化**してXXE攻撃を防ぐ
- **SSRF対策**：ユーザー指定URLはプライベートIP・内部ホストへのアクセスを拒否
- 信頼できないデータはデシリアライズしない
