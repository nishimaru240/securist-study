# 第5章: 入力処理

この章では、ユーザーやシステムからの入力データを**安全に処理する**ための設計を学ぶ。

- クライアント検証とサーバー検証の役割の違い
- 文字種・文字長の検証
- 文字エンコーディングの統一
- ファイルアップロード・XML・デシリアライズの危険と対策

---
# 入力検証の基本原則 ★

**入力値は「信頼できない」前提で扱う**

- ユーザーが送信するデータはすべて不正入力の可能性がある
- ネットワーク越しに来るデータは改ざんされている可能性もある

**検証の場所**

| 場所 | 役割 | 重要度 |
|------|------|--------|
| クライアント側（JS） | UX向上（即時フィードバック） | 補助的 |
| サーバー側 | セキュリティ保証 | **必須** |

→ **クライアント側の検証はバイパスできる。サーバー側で必ず再検証する**

---
# クライアント側検証がバイパスされる理由

**攻撃者はブラウザのJavaScriptを迂回してリクエストを送れる**

【迂回手段】
- ブラウザの開発者ツールでJSを無効化
- curlやBurp Suiteなどのツールで直接リクエストを送信
- JavaScriptを書き換えてバリデーションをスキップ

【設計上のルール】
- クライアント検証はUX目的と割り切る
- セキュリティ上の判断は**すべてサーバー側で行う**
- 「hidden フィールドの値は変更されない」という前提は禁物

---
# 文字種・文字長の検証 ★

**入力値は「許可する文字・長さ」を明示的に定義する（ホワイトリスト方式）**

| 項目 | 検証内容 | 例 |
|------|----------|----|
| 文字種 | 使用できる文字の種類 | 半角英数字のみ、日本語不可 |
| 文字長 | 最小・最大長 | 8〜128文字 |
| フォーマット | 入力値の形式 | メール形式、電話番号形式 |
| 数値範囲 | 数値の最小・最大値 | 0〜9999の整数 |

- 「禁止文字を除去する（ブラックリスト）」より「許可文字のみ受け付ける（ホワイトリスト）」が安全
- 最大長を設定しないとバッファオーバーフローやDoS攻撃の踏み台になる

---
# 文字エンコーディングの統一

**文字エンコーディング：文字をバイト列に変換する方式**

- 異なるエンコーディングが混在すると、文字化けや**セキュリティホール**が生まれる
- 例：Shift-JISの一部のバイト列がUTF-8の特殊文字と衝突してエスケープが無効化される

**統一設計**

- アプリ全体で **UTF-8** に統一する
- DBの文字コードもUTF-8（MySQL: `utf8mb4`）に統一する
- 入力受付時にエンコーディングを確認・変換する処理を入れる
- `Content-Type: text/html; charset=UTF-8` を必ずレスポンスに付ける

---
# 数値・整数の検証

**数値入力には型・範囲・桁数の検証を行う**

【ありがちな問題】
- 整数オーバーフロー：許容値を超えた数値を入れて計算結果が異常になる
- 負の数の悪用：商品個数に `-1` を入れて金額を不正に減らす
- 巨大な数値：`9999999999...` を渡して処理を止める（DoS）

【対策】
- 整数型・小数型・符号あり/なしを明示してパースする
- 業務上の最小値・最大値で範囲チェックする
- 入力値が想定の桁数を超えたら拒否する

---
# ファイルアップロードの危険と対策 ★

**アップロードファイルは複数の攻撃ベクターになる**

【危険なファイルの種類】
- PHPファイルなどのサーバーサイドスクリプト → アップロードして直接実行される
- HTMLファイル → XSSの踏み台
- 非常に大きなファイル → DoS攻撃

【安全な設計】
- 拡張子だけでなく **MIMEタイプ（ファイルの実際の内容）** で種類を判定する
- ブラウザが送る `Content-Type` は信用せず、**サーバー側でファイル先頭バイト（マジックナンバー）を確認**する
- アップロード先はWebサーバーの公開ディレクトリの**外**に置く（直接URL実行を防ぐ）
- ファイル名をサーバー側で付け直す（ユーザーが指定したファイル名を使わない）
- ファイルサイズの上限を設定する
- 画像はサーバー側で**再エンコード**して埋め込みスクリプトを除去する

---
# パス・トラバーサル攻撃

**パス・トラバーサル：`../` を使ってサーバーの意図しないファイルにアクセスする攻撃**

【攻撃例】
```
GET /file?name=../../etc/passwd
```
→ `/var/www/uploads/../../etc/passwd` = `/etc/passwd` を読み取られる

【対策】
- ファイル名に `../` や絶対パスが含まれないか検証する
- ファイルパスをアクセス許可ディレクトリと比較して範囲外を拒否する（パス正規化後に検証）
- ユーザー入力をファイルパスに直接使わず、**IDでマッピング**する設計にする

---
# XMLファイルの処理とXXE攻撃 ★

**XXE（XML External Entity）攻撃：外部エンティティ参照を悪用したサーバー内ファイルの読み取り攻撃**

【攻撃例（悪意あるXML）】
```xml
<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<root>&xxe;</root>
```
→ `/etc/passwd` の内容をXMLの値として出力させる

【対策】
- XMLパーサーの**外部エンティティ参照を無効化**する（DTD処理を無効化）
- 可能であればXMLではなくJSONを使用する
- 信頼できないソースからのXMLは受け付けない

---
# SSRF（サーバーサイドリクエストフォージェリ）★

**SSRF：サーバーに任意の宛先へリクエストを送らせる攻撃（OWASP Top 10 A10）**

【攻撃の例】
- ユーザーが URL を入力できるフォーム（画像取得・Webhook等）に、社内のIPアドレスを指定
  - 例：`http://169.254.169.254/...`（クラウドのメタデータエンドポイント）
- 外部からアクセスできない**社内システム・クラウドの認証情報**を窃取される

【対策】
- ユーザーが指定したURLへリクエストする前にホスト名を解決し、**プライベートIP・ループバックを拒否**する
- ホワイトリスト方式で**許可ドメインのみ**に通信先を絞る
- HTTPリダイレクトの追跡を無効化する（リダイレクト先が内部に向くのを防ぐ）

---
# デシリアライズ攻撃と対策

**デシリアライズ（Deserialization）：バイト列やJSON等のデータをオブジェクトに復元する処理**

**安全でないデシリアライズ（OWASP A08）：信頼できないデータをそのままデシリアライズすると任意コードが実行される**

【攻撃の仕組み】
- 攻撃者が細工したオブジェクトをシリアライズしてサーバーに送信
- デシリアライズ時にマジックメソッド（`__wakeup__` 等）が呼ばれてコードが実行される

【対策】
- 信頼できないソースからのシリアライズデータを**デシリアライズしない**
- デシリアライズ前にHMAC等で署名検証を行う
- Java の `ObjectInputStream` など危険なクラスの使用を避ける

---
# 第5章 まとめ ★

- 入力検証はサーバー側で必ず実施（クライアント側のみはバイパス可能）
- 検証は**ホワイトリスト方式**（許可する文字・長さ・形式を明示）
- 文字エンコーディングはアプリ・DB全体で **UTF-8 に統一**する
- ファイルアップロードはMIMEタイプ検証＋公開ディレクトリ外への保存
- XMLは**外部エンティティ参照を無効化**してXXE攻撃を防ぐ
- **SSRF対策**：ユーザー指定URLはプライベートIP・内部ホストへのアクセスを拒否
- 信頼できないデータはデシリアライズしない
