# 第7章: HTTPS・暗号化

この章では、通信を保護する**TLS/HTTPS**の仕組みと、安全な暗号設計の選び方を学ぶ。

- SSL/TLSの仕組みとハンドシェイク
- 証明書の種類と検証
- 安全な暗号アルゴリズムの選定
- プロトコルバージョンの選定
- HSTS による HTTPS 強制

---
# HTTPSとは ★

**HTTPS = HTTP + TLS（Transport Layer Security）**

- 通信内容を**暗号化**して盗聴・改ざんを防ぐ
- 接続先サーバーの**正当性を証明**してフィッシングを防ぐ
- 「鍵マーク」はサーバー証明書が有効であることを示す

**HTTPSがない場合のリスク**
- 通信内容（パスワード・セッションID等）が盗聴される
- 中間者攻撃（MITM）でデータを改ざんされる

→ **すべてのページをHTTPSで提供する**（ログインページだけでは不十分）

---
# 共通鍵暗号と公開鍵暗号 ★

**TLS は両方を組み合わせて使う**

| 種類 | 仕組み | 速度 | 用途 |
|------|--------|------|------|
| 共通鍵暗号 | 暗号化と復号に**同じ鍵**を使う（AES等） | 速い | データ本体の暗号化 |
| 公開鍵暗号 | 公開鍵で暗号化、秘密鍵で復号（RSA・ECDSA等） | 遅い | 鍵交換・電子署名 |

**TLSのハイブリッド方式**
- 通信開始時：公開鍵暗号で**共通鍵を安全に交換**する（鍵交換）
- 通信中：交換した共通鍵で**実データを高速に暗号化**する

> 鍵交換の流れ：①サーバーが証明書に公開鍵を入れてクライアントへ送る → ②**クライアントが**その公開鍵で共通鍵のもとになる情報を暗号化して送る → ③サーバーが秘密鍵で復号して共通鍵を生成。公開鍵は暗号化専用のため、傍受されても復号できない

→ 「速度」と「鍵配送問題の解決」を両立する設計

---
# TLS ハンドシェイクの仕組み ★

**接続確立時に暗号化の方式と鍵を安全に合意する手順**

1. Client Hello：クライアントが対応する暗号方式の一覧を送る
2. Server Hello：サーバーが使用する暗号方式を選択して証明書を送る
3. 証明書検証：クライアントがサーバー証明書の正当性を確認
4. 鍵交換：共通鍵を安全に生成・共有（TLS 1.3 では ECDHE：楕円曲線を使った鍵交換方式が必須）
5. 通信開始：合意した共通鍵で暗号化通信を開始

---
# サーバー証明書の種類

**証明書は「誰が認証されているか」の範囲によって3種類ある**

| 種類 | 確認内容 | 主な用途 |
|------|----------|----------|
| DV（ドメイン認証） | ドメイン所有者であること | 個人サイト・ブログ |
| OV（組織認証） | 組織の実在を確認 | 企業サイト |
| EV（拡張認証） | 組織の実在を厳格に確認 | 金融・ECサイト |

- 証明書はCA（認証局：Certificate Authority）が発行する
- **Let's Encrypt** は無料のDV証明書を自動発行するCA

---
# 証明書の検証とフィッシング対策

**証明書があっても「正しいサイト」とは限らない**

- フィッシングサイトも無料のDV証明書を取得できる
- 「鍵マーク＝安全」は誤解 → **ドメイン名を確認することが重要**

【証明書検証で確認すべき項目】
- CN（証明書に記載された代表ドメイン名）またはSAN（複数ドメインをまとめて記載できる拡張フィールド。現在はCN優先）が接続先と一致しているか
- 証明書の有効期限が切れていないか
- 信頼できるCAが発行しているか
- 失効していないか（CRL・OCSPで確認）

---
# 失効確認の仕組み：CRL・OCSP

**証明書は期限内でも秘密鍵漏洩等の理由で失効させることがある**

| 方式 | 仕組み | 欠点 |
|------|--------|------|
| CRL | CAが失効済み証明書すべてを一覧公開。ブラウザが丸ごとダウンロードして照合 | 無関係なサイトの証明書も含まれリストが肥大化しやすい |
| OCSP | ブラウザがCAに1件だけ問い合わせて確認 | CAへの遅延・CAダウン時に確認できない |
| OCSP Stapling | サーバーが事前にOCSP応答を取得してクライアントに渡す（現在の主流） | サーバー側の対応が必要 |

---
# 証明書に対する攻撃 ★

**証明書の仕組みそのものを狙う攻撃**

| 攻撃 | 内容 |
|------|------|
| CA 侵害 | 認証局が侵害されて攻撃者が任意ドメインの正規証明書を発行できてしまう |
| 偽証明書発行 | 本人確認不備をついて他人のドメイン用証明書を不正取得 |
| 中間CA 悪用 | 信頼された中間CAから不正な証明書を発行（DigiNotar事件など） |
| 秘密鍵漏洩 | サーバーの秘密鍵が流出 → 攻撃者が同じ証明書で偽サイトを構築 |
| ダウングレード攻撃 | 古いプロトコル・弱い暗号スイートに強制的に切替 |

---
# 偽証明書発行が成立する理由 ★

**DV証明書（ドメイン認証）の検証は機械的なため、迂回されることがある**

【典型的な検証方法と弱点】

- **メール認証**：`admin@example.com` 等の管理者用アドレス宛に確認メールを送る → そのメールアドレスを**攻撃者が乗っ取れる**と通過してしまう
- **DNS認証**：指定された TXT レコードをドメインに設定させる → 一時的な **DNS 乗っ取り・BGP ハイジャック**（経路情報を偽装する攻撃）で偽の応答を返せば通過
- **ファイル設置認証**：指定 URL にファイルを置かせる → **サブドメインテイクオーバー**（放置されたサブドメインのDNS設定を攻撃者が乗っ取る攻撃）で通過

→ DVは「ドメインを今この瞬間に支配しているか」しか見ない。**組織の実在性は確認しない**ため、機械的な検証の隙を突かれる

---
# 中間CAとは ★

**証明書の信頼は「ルートCA → 中間CA → サーバー証明書」の3段階で連鎖する**

```
ルートCA（厳重にオフライン保管、ブラウザに事前組込）
   ↓ 署名
中間CA（実際の発行業務を担当、オンライン稼働）
   ↓ 署名
サーバー証明書（あなたのサイトの証明書）
```

【中間CAが狙われやすい理由】

- **オンライン稼働**しているため攻撃対象になる（ルートはオフライン保管）
- 一度侵害されると**任意のドメインの正規証明書**を発行できてしまう（ブラウザは中間CAの署名を信頼するため）
- ルートCAより規模・予算の小さい組織が運営している場合がある
- 数が多く各国・各業界に複数存在する → 全体の防御を均一に保つのが難しい

→ ルートCAを直接守るより、中間CA侵害事案が現実には多い（DigiNotar・Comodo 事件）

---
# 証明書攻撃への対策

- **CAA レコード**（DNSに「このドメインの証明書を発行できるCA」を宣言）で不正発行を抑止
- **Certificate Transparency**（全証明書を公開ログに記録し不正発行を検知できる仕組み）の監視
- 秘密鍵漏洩時は**即座に失効申請**（CRL・OCSP に反映）し、新しい鍵ペアで再発行
- **HSTS** でHTTPダウングレードを防止、`includeSubDomains` でサブドメインも保護
- 重要サイトでは **証明書ピンニング**（公開鍵の**フィンガープリント**＝公開鍵を SHA-256 等でハッシュ化した短い識別子。長い鍵全体の代わりに比較できる、を事前登録）を検討
- 鍵長は **RSA 2048bit 以上 / ECDSA P-256 以上** を使う

---
# プロトコルバージョンの選定 ★

**古いTLSバージョンには脆弱性があるため使用禁止**

| バージョン | 状態 | 理由 |
|------------|------|------|
| SSL 3.0 | 禁止 | POODLE攻撃等の深刻な脆弱性 |
| TLS 1.0 | 禁止 | BEAST攻撃に弱い、PCI DSS非準拠 |
| TLS 1.1 | 禁止 | 非推奨アルゴリズムを含む |
| TLS 1.2 | 条件付き使用可 | 安全な暗号スイートの選択が必要 |
| TLS 1.3 | 推奨 | 速度・安全性ともに最良 |

→ **TLS 1.2 以上を許可し、TLS 1.3 を優先する設定にする**

---
# 古いTLSが禁止される理由

- **POODLE攻撃**（SSL 3.0 への攻撃）：暗号化されたCookieを1バイトずつ少しずつ復元できる手法。サーバーが古いバージョンに自動降格する仕組みを悪用するため、SSL 3.0 を許可するだけで他のバージョンを使っていても危険
- **BEAST攻撃**（TLS 1.0 への攻撃）：暗号文の規則性を突いて平文を推測する手法。ブラウザに罠ページを開かせて被害者の通信を盗聴する
- **TLS 1.1**：MD5やSHA-1など現代基準で安全でないアルゴリズムを使う設計が残っており、新しい攻撃手法に対応する仕組みがない

→ 古いバージョンを「念のため許可」しておくと攻撃者が自動降格させて利用するため、サーバー側で完全に無効化する必要がある

---
# 安全な暗号アルゴリズムの選定 ★

**暗号スイート：鍵交換・認証・暗号化・ハッシュの組み合わせ**

**推奨する設定**

| 用途 | 推奨アルゴリズム | 非推奨 |
|------|-----------------|--------|
| 鍵交換 | ECDHE（前方秘匿性あり） | RSA鍵交換 |
| 共通鍵暗号 | AES-256-GCM | RC4、DES、3DES |
| ハッシュ | SHA-256以上 | MD5、SHA-1 |

**前方秘匿性（Forward Secrecy）**：セッションごとに異なる鍵を使う仕組み。  
過去の通信データが将来解読されるリスクを低減する。

> **AES-256-GCM**：256ビット鍵のAESは現在の計算能力では総当たり不可能。GCM（Galois/Counter Mode）は暗号化と改ざん検知を同時に行う認証付き暗号モードで、盗聴だけでなく通信データの改ざんも検出できる
> **SHA-256以上**：ハッシュ値が256ビット以上。MD5（128bit）・SHA-1（160bit）は衝突攻撃（異なるデータから同じハッシュ値を意図的に生成する攻撃）で破られており使用禁止

---
# HSTS（HTTP Strict Transport Security）★

**HSTS：ブラウザに「このドメインは常にHTTPSで接続せよ」と命令する仕組み**

```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

- `max-age`：HTTPS強制を記憶する期間（秒）。1年（31536000秒）推奨
- `includeSubDomains`：サブドメインにも適用
- `preload`：ブラウザのHSTSプリロードリストに登録可能（プリロードリスト＝Chrome・Firefox等のブラウザに組み込まれたHTTPS強制ドメインの一覧。初回アクセスからHTTPSを強制でき、SSLストリッピングの隙をなくせる）

**HSTSがない場合の攻撃（SSLストリッピング）**
- HTTP でアクセスした最初のリクエストを中間者が傍受して HTTP のまま通信させる
- HSTS があれば初回からHTTPSを強制できる

---
# 混在コンテンツ（Mixed Content）の問題

**混在コンテンツ：HTTPSページの中でHTTPのリソース（画像・JS等）を読み込む問題**

- HTTP で読み込まれるリソースは暗号化されていない
- JSが混在するとページ全体のセキュリティが損なわれる（アクティブ混在コンテンツ）

【対策】
- すべてのリソースURLを `https://` に統一する
- 相対URL（`//example.com/img.png`）を使うとプロトコルを自動継承できる
- CSPの `upgrade-insecure-requests` ディレクティブでHTTPを自動HTTPSに昇格させる

---
# フィッシングサイトへの対抗策 ★

**「鍵マークがあれば安全」は通用しない。多層で防ぐ**

| 対策 | 効果 |
|------|------|
| EV/OV証明書の利用 | 組織実在性を証明書に記載（CAの厳格審査） |
| HSTSプリロード登録 | 初回アクセスからHTTPS強制、SSLストリッピング遮断 |
| FIDO2／WebAuthn | ドメイン名が署名に含まれ偽サイトでは認証不成立 |
| DMARC・SPF・DKIM | フィッシングメール送信元のなりすましを防ぐ |
| Certificate Transparency 監視 | 自社ドメインの不正証明書発行を検知 |
| ブランド類似ドメイン監視 | `examp1e.com` 等の紛らわしいドメイン取得を検知 |

→ サーバー側対策（証明書・HSTS）と利用者側対策（FIDO・ブラウザ警告）を組み合わせる

---
# DMARC・SPF・DKIM の仕組み ★

**メール送信元ドメインのなりすましを「DNS＋電子署名」で防ぐ3つの規格**

| 規格 | 仕組み | 何を検証するか |
|------|--------|----------------|
| SPF（Sender Policy Framework） | ドメイン所有者が**正当な送信サーバーのIPアドレス一覧**をDNSに公開 | 受信側が「送信元IPが許可リストに含まれるか」を照合 |
| DKIM（DomainKeys Identified Mail） | 送信側がメール本文・ヘッダーを**秘密鍵で電子署名**、公開鍵をDNSで公開 | 受信側が公開鍵で署名を検証 → 送信元正当性＋改ざん検知 |
| DMARC（Domain-based Message Authentication, Reporting and Conformance） | SPF/DKIM の**検証失敗時の処理ポリシー**（拒否／隔離／何もしない）をDNSで宣言 | 受信側が宣言通りに処理（拒否メールを送信元へレポートも可能） |

- SPF＋DKIM だけだと「失敗時に何をすべきか」が受信側任せ → **DMARC で送信側が明示する**
- 3つすべて設定して初めて「自社ドメインを騙ったメールは受信拒否される」状態が作れる

---
# 第7章 まとめ ★

- すべてのページを HTTPS で提供する（ログインページだけでは不十分）
- **TLS 1.2 以上を使用**し、TLS 1.3 を優先する。SSL・TLS 1.0/1.1 は無効化
- 暗号スイートは **ECDHE＋AES-GCM＋SHA-256以上** を選ぶ
- 証明書は有効期限・ドメイン一致・CA信頼性を確認する
- 証明書攻撃には **CAA・Certificate Transparency・証明書ピンニング** で備える
- **HSTS** を設定してSSLストリッピング攻撃を防ぐ
- 混在コンテンツ（HTTP リソース）をすべて HTTPS に統一する
- フィッシング対策は**サーバー側（HSTS・CT監視）＋利用者側（FIDO・MFA）**の多層で行う
