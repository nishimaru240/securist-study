# 第7章: HTTPS・暗号化

この章では、通信を保護する**TLS/HTTPS**の仕組みと、安全な暗号設計の選び方を学ぶ。

- SSL/TLSの仕組みとハンドシェイク
- 証明書の種類と検証
- 安全な暗号アルゴリズムの選定
- プロトコルバージョンの選定
- HSTS による HTTPS 強制

---
# HTTPSとは ★

**HTTPS = HTTP + TLS（Transport Layer Security）**

- 通信内容を**暗号化**して盗聴・改ざんを防ぐ
- 接続先サーバーの**正当性を証明**してフィッシングを防ぐ
- 「鍵マーク」はサーバー証明書が有効であることを示す

**HTTPSがない場合のリスク**
- 通信内容（パスワード・セッションID等）が盗聴される
- 中間者攻撃（MITM）でデータを改ざんされる

→ **すべてのページをHTTPSで提供する**（ログインページだけでは不十分）

---
# 共通鍵暗号と公開鍵暗号 ★

**TLS は両方を組み合わせて使う**

| 種類 | 仕組み | 速度 | 用途 |
|------|--------|------|------|
| 共通鍵暗号 | 暗号化と復号に**同じ鍵**を使う（AES等） | 速い | データ本体の暗号化 |
| 公開鍵暗号 | 公開鍵で暗号化、秘密鍵で復号（RSA・ECDSA等） | 遅い | 鍵交換・電子署名 |

**TLSのハイブリッド方式**
- 通信開始時：公開鍵暗号で**共通鍵を安全に交換**する（鍵交換）
- 通信中：交換した共通鍵で**実データを高速に暗号化**する

→ 「速度」と「鍵配送問題の解決」を両立する設計

---
# TLS ハンドシェイクの仕組み ★

**接続確立時に暗号化の方式と鍵を安全に合意する手順**

1. Client Hello：クライアントが対応する暗号方式の一覧を送る
2. Server Hello：サーバーが使用する暗号方式を選択して証明書を送る
3. 証明書検証：クライアントがサーバー証明書の正当性を確認
4. 鍵交換：共通鍵を安全に生成・共有（TLS 1.3 では ECDHE 必須）
5. 通信開始：合意した共通鍵で暗号化通信を開始

---
# サーバー証明書の種類

**証明書は「誰が認証されているか」の範囲によって3種類ある**

| 種類 | 確認内容 | 主な用途 |
|------|----------|----------|
| DV（ドメイン認証） | ドメイン所有者であること | 個人サイト・ブログ |
| OV（組織認証） | 組織の実在を確認 | 企業サイト |
| EV（拡張認証） | 組織の実在を厳格に確認 | 金融・ECサイト |

- 証明書はCA（認証局：Certificate Authority）が発行する
- **Let's Encrypt** は無料のDV証明書を自動発行するCA

---
# 証明書の検証とフィッシング対策

**証明書があっても「正しいサイト」とは限らない**

- フィッシングサイトも無料のDV証明書を取得できる
- 「鍵マーク＝安全」は誤解 → **ドメイン名を確認することが重要**

【証明書検証で確認すべき項目】
- コモンネーム（CN）またはSANが接続先ドメインと一致しているか
- 証明書の有効期限が切れていないか
- 信頼できるCAが発行しているか
- 失効していないか（CRL・OCSP で確認）

---
# プロトコルバージョンの選定 ★

**古いTLSバージョンには脆弱性があるため使用禁止**

| バージョン | 状態 | 理由 |
|------------|------|------|
| SSL 3.0 | 禁止 | POODLE攻撃等の深刻な脆弱性 |
| TLS 1.0 | 禁止 | BEAST攻撃に弱い、PCI DSS非準拠 |
| TLS 1.1 | 禁止 | 非推奨アルゴリズムを含む |
| TLS 1.2 | 条件付き使用可 | 安全な暗号スイートの選択が必要 |
| TLS 1.3 | 推奨 | 速度・安全性ともに最良 |

→ **TLS 1.2 以上を許可し、TLS 1.3 を優先する設定にする**

---
# 安全な暗号アルゴリズムの選定 ★

**暗号スイート：鍵交換・認証・暗号化・ハッシュの組み合わせ**

**推奨する設定**

| 用途 | 推奨アルゴリズム | 非推奨 |
|------|-----------------|--------|
| 鍵交換 | ECDHE（前方秘匿性あり） | RSA鍵交換 |
| 共通鍵暗号 | AES-256-GCM | RC4、DES、3DES |
| ハッシュ | SHA-256以上 | MD5、SHA-1 |

**前方秘匿性（Forward Secrecy）**：セッションごとに異なる鍵を使う仕組み。  
過去の通信データが将来解読されるリスクを低減する。

---
# HSTS（HTTP Strict Transport Security）★

**HSTS：ブラウザに「このドメインは常にHTTPSで接続せよ」と命令する仕組み**

```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

- `max-age`：HTTPS強制を記憶する期間（秒）。1年（31536000秒）推奨
- `includeSubDomains`：サブドメインにも適用
- `preload`：ブラウザのHSTSプリロードリストに登録可能

**HSTSがない場合の攻撃（SSLストリッピング）**
- HTTP でアクセスした最初のリクエストを中間者が傍受して HTTP のまま通信させる
- HSTS があれば初回からHTTPSを強制できる

---
# 混在コンテンツ（Mixed Content）の問題

**混在コンテンツ：HTTPSページの中でHTTPのリソース（画像・JS等）を読み込む問題**

- HTTP で読み込まれるリソースは暗号化されていない
- JSが混在するとページ全体のセキュリティが損なわれる（アクティブ混在コンテンツ）

【対策】
- すべてのリソースURLを `https://` に統一する
- 相対URL（`//example.com/img.png`）を使うとプロトコルを自動継承できる
- CSPの `upgrade-insecure-requests` ディレクティブでHTTPを自動HTTPSに昇格させる

---
# 第7章 まとめ ★

- すべてのページを HTTPS で提供する（ログインページだけでは不十分）
- **TLS 1.2 以上を使用**し、TLS 1.3 を優先する。SSL・TLS 1.0/1.1 は無効化
- 暗号スイートは **ECDHE＋AES-GCM＋SHA-256以上** を選ぶ
- 証明書は有効期限・ドメイン一致・CA信頼性を確認する
- **HSTS** を設定してSSLストリッピング攻撃を防ぐ
- 混在コンテンツ（HTTP リソース）をすべて HTTPS に統一する
