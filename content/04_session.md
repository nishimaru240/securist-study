# 第4章: セッション管理

この章では、ログイン状態を維持する**セッション**の仕組みと、セッションを狙った攻撃への対策を学ぶ。

- セッションIDの役割と生成
- Cookieの属性設計
- セッション固定攻撃・ハイジャック対策
- セッションタイムアウトとログアウト
- CSRF対策

---
# セッションIDの役割 ★

**HTTPはステートレス（状態を保持しない）プロトコル**

- リクエストごとに「誰か」をサーバーが識別できない
- ログイン後の状態を維持するために**セッションID**を使う

**仕組み**

1. ログイン成功 → サーバーがセッションIDを生成
2. セッションIDをCookieでブラウザに送付
3. 以降のリクエストにCookieが自動付与 → サーバーが識別

→ **セッションIDはログイン状態そのもの。盗まれると成りすまし可能**

---
# セッションIDの生成要件

**予測不可能なセッションIDを生成する**

- 十分な長さ（128ビット以上推奨）のランダム文字列
- 暗号論的疑似乱数生成器（CSPRNG）を使う
- 連番・ユーザーID・タイムスタンプ単体はNG（推測可能）

**フレームワークの標準機能を使う**

| 言語/FW | セッション生成 |
|---------|----------------|
| Java（Spring） | HttpSession（Tomcat管理） |
| Python（Django） | SESSION_KEY（デフォルトで安全） |
| PHP | session_start()（php.iniで強化） |

→ 自前で乱数生成せず、**フレームワーク標準に任せる**

---
# Cookie の属性設計 ★

**セッションIDを格納するCookieには必須属性がある**

| 属性 | 意味 | 設定すべき値 |
|------|------|--------------|
| HttpOnly | JavaScriptからアクセス不可 | 必ず付ける（XSS対策） |
| Secure | HTTPS通信時のみ送信 | 必ず付ける |
| SameSite | クロスサイトリクエスト時の送信制御 | Strict または Lax |
| Path | Cookieを送信するパスの範囲 | `/`（適切に絞る） |
| Expires/Max-Age | Cookieの有効期限 | セッション終了か適切な期限 |

---
# SameSite 属性の詳細

**SameSite：別ドメインからのリクエスト時にCookieを送信するかどうかを制御**

| 値 | 動作 |
|----|------|
| Strict | 同一サイトのリクエストのみCookieを送信 |
| Lax | GETによるトップレベルナビゲーションは許可、POSTは不可（デフォルト） |
| None | クロスサイトでも送信（Secure属性も必須） |

- CSRF対策として `Strict` または `Lax` を設定する
- `None` は外部サイト埋め込みが必要なケースのみ

---
# セッションIDの伝送方式

**セッションIDをクライアントに渡す方法**

| 方式 | 特徴 | 推奨 |
|------|------|------|
| Cookie | ブラウザが自動管理。属性で安全性を制御可能 | ✅ 推奨 |
| URL書き換え（URL Rewriting） | URLにセッションIDを埋め込む | ❌ 非推奨 |
| hidden パラメータ | フォームに埋め込む | ❌ 限定的 |

**URL書き換えの危険性**
- URLがブラウザ履歴・サーバーログ・リファラーに残り**漏洩しやすい**
- URLを共有した瞬間にセッションIDが流出する

→ セッションIDは **Cookie で管理する**

---
# セッション固定攻撃（Session Fixation）

**セッション固定攻撃：攻撃者が用意したセッションIDをユーザーに使わせる攻撃**

【攻撃の流れ】
1. 攻撃者が事前にセッションIDを取得（例：URLパラメーターで送る）
2. 被害者がそのセッションIDでログイン
3. 攻撃者は同じセッションIDでログイン済み状態にアクセスできる

【対策】
- **ログイン成功後に必ず新しいセッションIDを発行する**（セッションID再生成）
- セッションIDをURLパラメーターに含めない（Cookieのみで管理）

**セッションIDを再生成すべきタイミング**
- ログイン成功時（最重要）
- 権限が変わったとき（管理者権限への昇格等）
- パスワード変更などの重要操作の直後

---
# セッションハイジャック対策

**セッションハイジャック：有効なセッションIDを盗んで成りすます攻撃**

【主な盗聴・盗取手段】
- HTTP通信の盗聴（HTTPSで防ぐ）
- XSSによるCookieの窃取（HttpOnly属性で防ぐ）
- 中間者攻撃（HTTPS＋HSTS で防ぐ）

【追加の対策設計】
- セッションIDとIPアドレスを紐付け、変化時に再認証を要求する
- User-Agentの急激な変化を検知してセッションを無効化する

---
# セッションタイムアウト設計

**タイムアウトの種類**

| 種類 | 意味 | 推奨設定 |
|------|------|----------|
| アイドルタイムアウト | 最後の操作から一定時間で失効 | 15〜30分（金融系は短く） |
| 絶対タイムアウト | ログインからの絶対時間で失効 | 8〜24時間 |

**設計のポイント**
- タイムアウト後はサーバー側でセッションを削除する（Cookie削除だけでは不十分）
- タイムアウト前に「セッションが切れます」警告を出してUXに配慮する

---
# ログアウト機能の設計

**ログアウトは「Cookieを削除するだけ」では不十分**

【正しいログアウト処理】
1. サーバー側のセッションデータを完全に削除する
2. `Set-Cookie` でCookieの `Max-Age=0` を設定して削除指示を送る
3. キャッシュをクリアするレスポンスヘッダーを付ける

**SSO環境ではシングルログアウト（SLO）も実装する**
- RP側だけでなく、IdP（認証サービス）側のセッションも無効化する
- 連携する他のRPすべてもログアウトさせる

---
# CSRF（クロスサイトリクエストフォージェリ）★

**CSRF：ログイン済みユーザーに意図しないリクエストを実行させる攻撃**

【攻撃の流れ】
1. 被害者が正規サイトにログイン済み（Cookieあり）
2. 攻撃者の罠サイトを訪問する
3. 罠サイトから正規サイトへのPOSTリクエストが自動送信される
4. Cookieが自動付与されるため、サーバーは正当なリクエストと誤認する

→ **被害者が知らないうちにパスワード変更・送金等が実行される**

---
# CSRF 対策：トークン方式 ★

**CSRFトークン：サーバーが生成したランダムな値をフォームに埋め込む方式**

【仕組み】
1. フォーム表示時にサーバーがランダムなトークンを発行
2. HTMLの hidden フィールドにトークンを埋め込む
3. POST送信時にトークンをサーバーが検証
4. トークンが一致しない場合はリクエストを拒否

- 罠サイトはCSRFトークンの値を知ることができないため攻撃が無効化される
- セッションごとに新しいトークンを発行する

**CSRFを防ぐ設計原則：副作用のある操作は必ずPOSTにする**
- GETでデータ更新・削除を行う設計はCSRFに対して脆弱になりやすい
- GETは「冪等（何度実行しても結果が同じ）」な参照系のみに使う

---
# CSRF 対策：SameSite 属性 ★

**SameSite=Strict/Lax を設定するだけでCSRFをほぼ防げる**

- `Strict`：クロスサイトのリクエストにはCookieを一切送らない → CSRF完全防止
- `Lax`：POSTへのクロスサイトリクエストにはCookieを送らない → 一般的なCSRF防止

**推奨設計**

- SameSite 属性＋CSRFトークンの**両方を使う**（多層防御）
- 古いブラウザはSameSiteに非対応のためトークン方式が保険になる

---
# 第4章 まとめ ★

- セッションIDは **128ビット以上のCPRNG** で生成し、フレームワーク任せにする
- CookieはHttpOnly・Secure・SameSiteをすべて設定する
- **ログイン後にセッションIDを再生成**してセッション固定攻撃を防ぐ
- タイムアウトはサーバー側でセッションを削除（Cookie削除だけでは不十分）
- CSRF対策は**トークン方式＋SameSite属性**の多層防御
