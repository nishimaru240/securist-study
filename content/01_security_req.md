# 第1章: セキュリティ要件

この章では、セキュアなWebアプリケーションを設計するための**基礎的な考え方**と、業界標準の要件フレームワークを学ぶ。

- セキュリティの3原則（CIA）
- OWASP Top 10
- PCI DSS

---
# セキュリティ要件とは

**システムが「何を守るか」「どう守るか」を定義したもの**

- 機能要件（何ができるか）とは別に定義する必要がある
- 開発の後付けではなく、**設計段階から組み込む**ことが重要
- 要件が曖昧だと、脆弱性の見落としや手戻りが発生する

---
# セキュリティの3原則（CIA）★

| 原則 | 英語 | 意味 |
|------|------|------|
| 機密性 | Confidentiality | 許可された人だけがアクセスできる |
| 完全性 | Integrity | 情報が改ざんされていない |
| 可用性 | Availability | 必要なときにシステムを利用できる |

**3つすべてのバランスを取ることが重要**

---
# なぜ設計段階でセキュリティ要件が必要か

**後から対策すると、コストが10〜100倍になる**（NIST試算）

- 開発中の修正：数時間
- テスト後の修正：数日
- リリース後の修正：数週間〜数ヶ月＋信頼失墜

→ **セキュリティは最初から設計に組み込む（Secure by Design：要件定義・設計フェーズからセキュリティを考慮する開発アプローチ）**

---
# OWASPとは

**Open Web Application Security Project**

- Webアプリケーションセキュリティの向上を目的とした非営利団体
- セキュリティガイドラインや脆弱性情報を無償で公開
- 世界中の企業・政府機関のセキュリティ基準として採用されている

代表的な成果物: **OWASP Top 10**（最も重大なWebリスク10項目）

---
# OWASP Top 10（2021年版）前半 ★

| 順位 | カテゴリ |
|------|----------|
| A01 | アクセス制御の不備 |
| A02 | 暗号化の失敗 |
| A03 | インジェクション（SQL・OSコマンド等） |
| A04 | 安全でない設計 |
| A05 | セキュリティの設定ミス |

---
# OWASP Top 10（2021年版）後半 ★

| 順位 | カテゴリ |
|------|----------|
| A06 | 脆弱で古いコンポーネントの使用 |
| A07 | 識別と認証の失敗 |
| A08 | ソフトウェアとデータの整合性の失敗 |
| A09 | セキュリティログとモニタリングの失敗 |
| A10 | サーバーサイドリクエストフォージェリ（SSRF） |

> **SSRF**：攻撃者がサーバーに任意のURLへリクエストを送らせる攻撃。内部ネットワークのサービスやクラウドのメタデータAPIに不正アクセスされる危険がある。対策は入力URLのホワイトリスト検証・内部IPアドレスへのアクセス禁止。

---
# SSRF 攻撃の流れ ★

**例：「URLで画像を取得する機能」を悪用するケース**

1. `?url=` に任意のURLを渡せると気づく（脆弱性の発見）
2. `?url=http://192.168.1.1/` などで内部ホストの存在を確認（内部探索）
3. `?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/` を送信
   → サーバーがAWSメタデータAPIを代理リクエスト → **一時AWSアクセスキーが返る**
4. 取得したキーでS3・DBに不正アクセスまたは管理画面を操作（権限昇格・横展開）

**攻撃が成立する理由**：攻撃者は内部ネットワークに直接届かないが、Webサーバーを踏み台にすることで到達できる

---
# PCI DSSとは

**Payment Card Industry Data Security Standard**（v4.0、2022年策定）

- クレジットカード情報を扱う事業者が遵守すべきセキュリティ基準
- カード会社（Visa、Mastercard等）が共同で策定
- Webアプリケーションにも適用される要件が多い

**OWASP Top 10への対応がPCI DSS準拠にも直結する**

---
# PCI DSS の主な要件（Webアプリ関連）

- 安全なシステムとアプリケーションの開発・保守
- ネットワーク越しのカードデータ送信を暗号化（HTTPS）
- すべてのシステムのアクセス制御
- 脆弱性管理プログラムの維持
- 定期的なセキュリティテストの実施

---
# Webアプリケーションのセキュリティ要件書

**「脆弱性診断士スキルマッププロジェクト」が公開する要件書**

- SecuriST試験の出題根拠となるドキュメント
- セキュリティ要件を網羅的にリスト化
- 発注者・設計者・開発者が共通言語として使える

→ この試験で学ぶ内容はこの要件書に基づいている

# 第1章 まとめ

- セキュリティ要件は**設計段階から**定義する
- CIAの3原則（機密性・完全性・可用性）が基礎
- **OWASP Top 10**：Webアプリの代表的リスク10項目
- **PCI DSS**：カード情報を扱う事業者の準拠基準
- 後付け対策はコストが跳ね上がる → Secure by Design
