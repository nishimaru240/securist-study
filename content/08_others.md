# エラーメッセージのハンドリング ★

**エラーメッセージは攻撃者への情報提供になる**

【情報漏洩につながるエラー例】
- スタックトレース（フレームワーク・ファイルパス・DBスキーマが露出）
- SQL文そのもの（テーブル名・カラム名が露出）
- 「メールアドレスは登録済みです」→ アカウントの存在が判明

【安全な設計】
- ユーザーには**汎用的なエラーメッセージ**だけを表示する
  - 例：「処理に失敗しました。しばらくしてから再度お試しください」
- 詳細なエラー情報はサーバー側のログにだけ記録する
- 本番環境でデバッグモードを有効にしない

---
# エラーページの設計

**カスタムエラーページで情報漏洩を防ぐ**

| HTTPステータス | 表示内容 |
|----------------|----------|
| 400 Bad Request | 不正なリクエストです |
| 403 Forbidden | アクセス権限がありません |
| 404 Not Found | ページが見つかりません |
| 500 Internal Server Error | サーバーエラーが発生しました |

- フレームワークのデフォルトエラーページはバージョン情報を含む場合がある → 必ずカスタム化する
- 404と403を**意図的に区別しない**ケースもある（存在の有無を隠す）

# ログ記録の設計 ★

**セキュリティログは「何が起きたか」を後から追跡するための記録**

**記録すべきイベント**
- ログイン成功・失敗（IPアドレス・タイムスタンプ）
- パスワード変更・リセット
- 権限変更・管理者操作
- 異常な操作（短時間の大量リクエスト等）

**記録してはいけない情報**
- パスワード・クレジットカード番号などの機密情報
- セッションID（ログが漏洩するとセッションハイジャックに使われる）

---
# ログの保護と管理

**ログ自体がセキュリティ上の重要資産**

【ログの改ざん対策】
- アプリサーバーとは別の**ログ専用サーバー**に転送する
- ログへの書き込みは追記のみ許可し、削除・変更を禁止する

【ログの保存期間】
- PCI DSS：最低12ヶ月（3ヶ月分はオンライン参照可能な状態）
- セキュリティインシデント調査に必要な期間を考慮して設定する

【ログの監視】
- 異常なパターン（連続ログイン失敗等）をリアルタイムで検知するアラートを設定する

# CAPTCHA ★

**CAPTCHA（Completely Automated Public Turing test to tell Computers and Humans Apart）：**  
**人間とボットを区別するためのチャレンジ認証**

| 種類 | 例 | 特徴 |
|------|-----|------|
| 画像認識 | 「信号機を選択」 | 古典的。視覚障害者に不便 |
| テキスト入力 | ゆがんだ文字を入力 | OCRで突破されやすくなった |
| 行動分析 | reCAPTCHA v3 | スコアで自動判定、UIなし |

**利用場面**
- ログインフォームへのブルートフォース対策
- 会員登録フォームのスパム防止
- パスワードリセット要求の大量送信防止

---
# レート制限（Rate Limiting）

**レート制限：単位時間あたりのリクエスト数を制限する仕組み**

- ブルートフォース攻撃・スクレイピング・DoS攻撃の抑止に有効
- アカウント単位・IP単位・APIキー単位など複数の軸で制限する

【主な制御方式】

| 方式 | 仕組み |
|------|--------|
| 固定ウィンドウ | 1分間にN回までなど、時間枠で制限 |
| トークンバケット | 一定速度で補充されるトークンを消費 |

【設計のポイント】
- 制限超過時は **HTTP 429 Too Many Requests** を返す
- ログイン・パスワードリセット等の重要エンドポイントは**特に厳しく**制限する

---
# クリックジャッキング対策 ★

**クリックジャッキング：透明なiframeで正規サイトを重ね、見えないボタンをクリックさせる攻撃**

【攻撃の仕組み】
1. 攻撃者サイトが罠ページを用意
2. 正規サイトを透明なiframeで上に重ねる
3. ユーザーが「ボタンを押した」つもりで正規サイトの操作を実行してしまう

【対策：X-Frame-Options ヘッダー】
```
X-Frame-Options: DENY
```
- `DENY`：全iframeへの埋め込みを禁止
- `SAMEORIGIN`：同一オリジンのみ許可

【モダンな対策：CSP の frame-ancestors】
```
Content-Security-Policy: frame-ancestors 'none'
```

# キャッシュ制御

**ブラウザキャッシュに機密情報が残ると情報漏洩につながる**

【問題のあるシナリオ】
- 共有PCでログインして離れた後、履歴から機密ページが見られる
- プロキシサーバーが認証後のレスポンスをキャッシュして別ユーザーに返す

【キャッシュを禁止するレスポンスヘッダー】
```
Cache-Control: no-store, no-cache
Pragma: no-cache
```

- ログイン後のページ・個人情報・取引履歴等には必ず設定する
- 静的リソース（CSS・画像等）はキャッシュを許可してよい

# 脆弱性診断とセキュリティテスト

**設計だけでなく「実装後の検証」もセキュリティ品質を支える**

| 種類 | 内容 | タイミング |
|------|------|------------|
| SAST（静的解析） | ソースコードを解析して脆弱性を検出 | 開発中・CIに組込み |
| DAST（動的解析） | 稼働中のアプリにリクエストを送って脆弱性を検出 | テスト・ステージング |
| ペネトレーションテスト | 専門家が攻撃者視点で実際に侵入を試みる | リリース前・定期 |
| 依存ライブラリ診断（SCA） | 使っているOSSライブラリの既知脆弱性を検出 | CI・依存追加時 |

→ **設計時のレビュー＋実装後の自動診断＋定期的なペンテスト**を組み合わせる

---
# セキュリティヘッダーの総まとめ ★

**設定すべきレスポンスヘッダー一覧**

| ヘッダー | 推奨値 | 目的 |
|----------|--------|------|
| `Content-Security-Policy` | `default-src 'self'` | XSS防止 |
| `X-Frame-Options` | `DENY` | クリックジャッキング防止 |
| `X-Content-Type-Options` | `nosniff` | MIMEスニッフィング防止 |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` | HTTPS強制 |
| `Cache-Control` | `no-store`（機密ページ） | 情報漏洩防止 |
| `Referrer-Policy` | `no-referrer` または `strict-origin` | リファラー漏洩防止 |
