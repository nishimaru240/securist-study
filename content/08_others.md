# エラーメッセージのハンドリング ★

**エラーメッセージは攻撃者への情報提供になる**

【情報漏洩につながるエラー例】
- スタックトレース（エラー発生時にどのファイルの何行目でどの関数が呼ばれていたかを示す処理経路の記録。フレームワーク・ファイルパス・DBスキーマが露出する）
- SQL文そのもの（テーブル名・カラム名が露出）
- 「メールアドレスは登録済みです」→ アカウントの存在が判明

【安全な設計】
- ユーザーには**汎用的なエラーメッセージ**だけを表示する
  - 例：「処理に失敗しました。しばらくしてから再度お試しください」
- 詳細なエラー情報はサーバー側のログにだけ記録する
- 本番環境でデバッグモード（フレームワークの設定ファイルや環境変数で開発中に有効にする設定。エラーの詳細・変数の中身・設定値等をブラウザに直接表示する）を有効にしない

---
# エラーページの設計

**カスタムエラーページで情報漏洩を防ぐ**

| HTTPステータス | 表示内容 |
|----------------|----------|
| 400 Bad Request | 不正なリクエストです |
| 403 Forbidden | アクセス権限がありません |
| 404 Not Found | ページが見つかりません |
| 500 Internal Server Error | サーバーエラーが発生しました |

- フレームワークのデフォルトエラーページはバージョン情報を含む場合がある → 必ずカスタム化する
- 404と403を**意図的に区別しない**ケースもある（存在の有無を隠す）

---
# ログ記録の設計 ★

**セキュリティログは「何が起きたか」を後から追跡するための記録**

**記録すべきイベント**
- ログイン成功・失敗（IPアドレス・タイムスタンプ）
- パスワード変更・リセット
- 権限変更・管理者操作
- 異常な操作（短時間の大量リクエスト等）

**記録してはいけない情報**
- パスワード・クレジットカード番号などの機密情報
- セッションID（ログが漏洩するとセッションハイジャックに使われる）

---
# ログの保護と管理

**ログ自体がセキュリティ上の重要資産**

【ログの改ざん対策】
- アプリサーバーとは別の**ログ専用サーバー**に転送する
- ログへの書き込みは追記のみ許可し、削除・変更を禁止する

【ログの保存期間】
- PCI DSS（Payment Card Industry Data Security Standard：クレジットカード情報を扱う事業者が準拠すべきセキュリティ基準）：最低12ヶ月（3ヶ月分はオンライン参照可能な状態）
- セキュリティインシデント調査に必要な期間を考慮して設定する

【ログの監視】
- 異常なパターン（連続ログイン失敗等）をリアルタイムで検知するアラートを設定する

# ユーザーへの通知 ★

**不正利用の早期検知は「ユーザー本人に気付かせる」設計が肝心**

**通知すべきイベント**

| イベント | 通知タイミング |
|----------|----------------|
| パスワード変更・リセット完了 | 完了直後にメール |
| 登録メールアドレス・電話番号の変更 | 旧アドレスにも必ず通知 |
| 普段と異なる端末・地域からのログイン | ログイン成功時にメール |
| 二要素認証の有効化・無効化 | 完了直後にメール |
| 高額決済・送金等の重要操作 | 完了直後にメール／プッシュ |

- 通知メールには**「身に覚えがない場合の問い合わせ窓口」**を明記する
- 通知自体を**ロケールに合わせた言語**で送る（フィッシングと区別しやすくする）

# 暗号アルゴリズムと乱数 ★

**用途ごとに適切な暗号方式を選ぶ。「自作暗号」は厳禁**

| 用途 | 推奨アルゴリズム | 非推奨 |
|------|------------------|--------|
| 共通鍵暗号（データ暗号化） | AES-256-GCM、ChaCha20-Poly1305 | DES、3DES、RC4 |
| 公開鍵暗号（鍵交換・署名） | ECDSA（P-256以上）、RSA 2048bit以上 | RSA 1024bit |
| ハッシュ（改ざん検知） | SHA-256以上、SHA-3 | MD5、SHA-1 |
| パスワードハッシュ | Argon2、bcrypt、PBKDF2 | SHA-256単体 |
| メッセージ認証（MAC） | HMAC-SHA-256 以上 | 自前のXOR等 |

- アルゴリズムは**標準ライブラリ・OpenSSL等の実績ある実装**を使う
- 暗号鍵は**コードに埋め込まず**、KMS（Key Management Service）や環境変数で管理する

---
# 疑似乱数生成器 ★

**乱数には「予測可能な乱数」と「予測不能な乱数」がある**

| 種類 | 用途 | 例 |
|------|------|----|
| 通常の擬似乱数（PRNG） | 統計・ゲーム・UI演出 | `Math.random()`、`rand()` |
| 暗号論的疑似乱数（CSPRNG） | セキュリティ用途すべて | `crypto.randomBytes()`、`SecureRandom` |

**CSPRNG を使うべき場面**

- セッションID・CSRFトークン・パスワードリセットトークン
- 初期パスワード・APIキー・APIシークレット
- salt・IV（初期化ベクトル）・ナンス

> **PRNG が危険な理由**：内部状態が単純で過去の出力から次の値を予測できる。攻撃者にセッションIDを推測されると成りすましが成立する

→ **「乱数が必要」と思ったらまず CSPRNG を使う**（迷ったらCSPRNG）

# CAPTCHA ★

**CAPTCHA（Completely Automated Public Turing test to tell Computers and Humans Apart）：**  
**人間とボットを区別するためのチャレンジ認証**

| 種類 | 例 | 特徴 |
|------|-----|------|
| 画像認識 | 「信号機を選択」 | 古典的。視覚障害者に不便 |
| テキスト入力 | ゆがんだ文字を入力 | OCRで突破されやすくなった |
| 行動分析 | reCAPTCHA v3 | マウスの動きや操作履歴をスコア化して自動判定、UIなし |

**利用場面**
- ログインフォームへのブルートフォース対策
- 会員登録フォームのスパム防止
- パスワードリセット要求の大量送信防止

---
# レート制限（Rate Limiting）

**レート制限：単位時間あたりのリクエスト数を制限する仕組み**

- ブルートフォース攻撃・スクレイピング・DoS攻撃の抑止に有効
- アカウント単位・IP単位・APIキー単位など複数の軸で制限する

【主な制御方式】

| 方式 | 仕組み |
|------|--------|
| 固定ウィンドウ | 1分間にN回までなど、時間枠で制限 |
| トークンバケット | 一定速度で補充されるトークンを消費 |

【設計のポイント】
- 制限超過時は **HTTP 429 Too Many Requests** を返す
- ログイン・パスワードリセット等の重要エンドポイントは**特に厳しく**制限する

---
# クリックジャッキング対策 ★

**クリックジャッキング：透明なiframeで正規サイトを重ね、見えないボタンをクリックさせる攻撃**

【対策：X-Frame-Options ヘッダー】
```
X-Frame-Options: DENY
```
- `DENY`：全iframeへの埋め込みを禁止
- `SAMEORIGIN`：同一オリジンのみ許可

【モダンな対策：CSP の frame-ancestors】
```
Content-Security-Policy: frame-ancestors 'none'
```
- iframe内への描画自体をブロック

---
# CORS と Access-Control-Allow-Origin ★

**CORS（Cross-Origin Resource Sharing）：別オリジンからのAPI呼び出しを制御する仕組み**

- ブラウザは**同一オリジンポリシー**（同じプロトコル・ドメイン・ポートでないとJSがレスポンスを読めない）で保護されている
- API を別ドメインのフロントから呼ぶ場合、サーバーが `Access-Control-Allow-Origin` で許可を明示する必要がある

**主なCORS関連レスポンスヘッダー**

| ヘッダー | 役割 |
|----------|------|
| `Access-Control-Allow-Origin` | 許可するオリジンを指定 |
| `Access-Control-Allow-Credentials` | Cookie・認証情報の送信を許可するか |
| `Access-Control-Allow-Methods` | 許可するHTTPメソッド |
| `Access-Control-Allow-Headers` | 許可するリクエストヘッダー |

---
# CORS 設定の典型的なミス

**設計を間違えると CSRF と同等の被害が出る**

【危険な設定】
- `Access-Control-Allow-Origin: *` ＋ `Allow-Credentials: true` → **どんなサイトからも認証付きAPIを叩ける**（仕様上 `*` と `true` の併用はブラウザが拒否するが、リクエストOriginをそのまま反映する実装で同等の事故が起きる）
- リクエスト元の `Origin` ヘッダーを**無検証で** `Allow-Origin` に転記する → 複数オリジンに動的対応するつもりが**全オリジン許可**と同義になる

【安全な設計】
- 許可オリジンは**ホワイトリスト**で完全一致検証
- 公開APIは `Allow-Credentials: false` にし、認証はトークンで行う
- プリフライト（**OPTIONS**リクエスト：本リクエストを送る前にブラウザが自動で送る事前確認。`PUT`/`DELETE` や `Authorization` ヘッダー、`application/json` 等を含む場合に発生し、サーバーが「許可するメソッド・ヘッダー」を返すことで本リクエストが許される）でも同じ検証を実施

# キャッシュ制御

**ブラウザキャッシュに機密情報が残ると情報漏洩につながる**

【問題のあるシナリオ】
- 共有PCでログインして離れた後、履歴から機密ページが見られる
- プロキシサーバーが認証後のレスポンスをキャッシュして別ユーザーに返す

【キャッシュを禁止するレスポンスヘッダー】
```
Cache-Control: no-store, no-cache
Pragma: no-cache
```

| 値 | 意味 |
|---|---|
| `no-store` | キャッシュに一切保存しない（最も強力。機密ページに必須） |
| `no-cache` | 保存はするが毎回サーバーに再検証してから使う（変更なしなら304でキャッシュ使用） |
| `Pragma: no-cache` | HTTP/1.0時代の旧ヘッダー。古いプロキシとの互換性のために併記する |

- ログイン後のページ・個人情報・取引履歴等には必ず設定する
- 静的リソース（CSS・画像等）はキャッシュを許可してよい

# 言語・フレームワーク・ライブラリの選定 ★

**「使うものを選ぶ」段階からセキュリティが始まる**

**選定時にチェックする項目**

- **メンテナンス状況**：最終コミット日、最新リリース、アクティブなコミッタ数
- **既知脆弱性の履歴**：CVE 件数、対応スピード（修正リリースまでの日数）
- **サポート期限（EOL）**：言語・フレームワークのバージョンが保守対象内か
- **コミュニティ規模**：採用実績の多さ、情報入手のしやすさ
- **セキュリティドキュメント**：公式に脆弱性報告窓口があるか

→ **マイナーすぎる OSS は脆弱性発見・修正が遅れる傾向**

---
# 言語環境のセキュリティ設定 ★

**フレームワーク・言語ランタイムには「危険な機能」が標準で有効な場合がある**

| 環境 | 確認すべき設定 |
|------|----------------|
| PHP | `expose_php=Off`、`display_errors=Off`、`allow_url_include=Off` |
| Java | デシリアライズフィルタの設定、`SecurityManager` 廃止後の代替 |
| Python | `DEBUG=False`（本番）、`pickle` の使用回避 |
| Node.js | `--inspect` を本番で開かない、依存の `npm audit` |
| Ruby on Rails | `config.force_ssl=true`、`secret_key_base` のローテーション |

- 本番・ステージング・開発の**環境別に設定を分離**する
- 設定値は**コードに直書きせず**、環境変数やシークレットマネージャで管理

# 開発プロセス・パイプラインのセキュリティ ★

**「設計後にレビュー」では遅い。開発の全工程に組み込む（Secure SDLC）**

| フェーズ | セキュリティ施策 |
|----------|------------------|
| 要件定義 | セキュリティ要件の明文化（OWASP ASVS 等を参照） |
| 設計 | 脅威モデリング（STRIDE）、設計レビュー |
| 実装 | コーディング規約、SAST 自動実行 |
| ビルド | SCA で OSS 脆弱性検出、SBOM 生成 |
| デプロイ | DAST、シークレットスキャン、署名付きアーティファクト |
| 運用 | WAF・SIEM 監視、定期的なペネトレーションテスト |

> **STRIDE**：Microsoft 提唱の脅威分類フレームワーク（Spoofing/Tampering/Repudiation/Information disclosure/Denial of service/Elevation of privilege）

---
# パイプライン保護とシークレット管理

**CI/CD パイプラインも攻撃対象**

- パイプライン定義（`.github/workflows/` 等）に**書き込み権限を持つ人を最小化**
- 外部からのPRに対しては**シークレットを渡さない**設定（pull_request_target の悪用防止）
- ビルド成果物に**署名**を付与（改ざん検知、サプライチェーン攻撃対策）

**シークレット管理**

- API キー・パスワード・秘密鍵を**コード・Gitにコミットしない**（gitleaks 等で自動検査）
- **シークレットマネージャ**（AWS Secrets Manager、HashiCorp Vault）で集中管理
- 漏洩時に即座にローテーションできる仕組みを用意

# 用意すべきドキュメント ★

**設計だけでなく「運用・有事」のドキュメントもセキュリティの一部**

| ドキュメント | 目的 |
|--------------|------|
| セキュリティ要件書 | 守るべき資産・脅威・要件を明文化 |
| 脅威モデル | システム構成図と想定攻撃シナリオ |
| 設計書（セキュリティ章） | 認証・認可・暗号・ログの設計判断 |
| 運用手順書 | 鍵ローテーション・証明書更新・パッチ適用手順 |
| インシデント対応手順書 | 検知・初動・連絡・収束・再発防止のフロー |
| 脆弱性管理台帳 | 検出脆弱性の対応状況・期日・担当 |
| プライバシーポリシー | 個人情報の取り扱い（法令対応） |

→ ドキュメントは**最新状態を保つ**ことが重要（古い手順書は混乱の元）

---
# 脆弱性診断とセキュリティテスト

**設計だけでなく「実装後の検証」もセキュリティ品質を支える**

| 種類 | 内容 | タイミング |
|------|------|------------|
| SAST（Static Application Security Testing・静的解析） | ソースコードを解析して脆弱性を検出 | 開発中・CIに組込み |
| DAST（Dynamic Application Security Testing・動的解析） | 稼働中のアプリにリクエストを送って脆弱性を検出 | テスト・ステージング |
| ペネトレーションテスト | 専門家が攻撃者視点で実際に侵入を試みる | リリース前・定期 |
| SCA（Software Composition Analysis・依存ライブラリ診断） | 使っているOSSライブラリの既知脆弱性を検出 | CI・依存追加時 |

→ **設計時のレビュー＋実装後の自動診断（SAST・DAST）＋定期的なペネトレーションテスト**を組み合わせる

# セキュリティヘッダーの総まとめ ★

**設定すべきレスポンスヘッダー一覧**

| ヘッダー | 推奨値 | 目的 |
|----------|--------|------|
| `Content-Security-Policy` | `default-src 'self'` | XSS防止 |
| `X-Frame-Options` | `DENY` | クリックジャッキング防止 |
| `X-Content-Type-Options` | `nosniff` | MIMEスニッフィング防止 |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` | HTTPS強制 |
| `Cache-Control` | `no-store`（機密ページ） | 情報漏洩防止 |
| `Referrer-Policy` | `no-referrer` または `strict-origin` | リファラー漏洩防止 |

> **リファラー**（Referer）：リンクをクリックしてページ遷移する際にブラウザが自動送信する「遷移元URL」を示すHTTPヘッダー。URLにセッションIDやトークンが含まれていると外部サイトに漏洩する。`no-referrer` は一切送信しない、`strict-origin` はドメイン名のみ送信してパスやクエリを隠す
