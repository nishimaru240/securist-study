# 試験出題情報

SecuriST 認定セキュアWebアプリケーション設計士の試験形式・出題傾向・典型問題例をまとめる。
公式サンプル問題は試験規約で外部公開が禁止されているため、合格体験記と徳丸本ベースから推測した典型例を掲載する。

## 出題傾向

合格体験記から判明している傾向：

- **公式テキストの内容が満遍なく出題される**（特定領域に偏らない）
- **単純な用語暗記では通らない**：「攻撃手法がどんなもので、ソフトウェアにどんな設計をすれば防げるか」を整理して理解している必要がある
- 公式テキストを3周精読しても「迷う問題が何問かあった」と評される難易度
- コードや設定例を提示して「脆弱性を特定」「適切な対策を選択」させる形式が想定される

### 特に精読を推奨される領域

- セキュリティ要件
- ログイン機能・パスワード入力フォーム設計
- パスワード保管・リセット機能設計
- ユーザ管理・セッション管理
- Cookie設定、CSRF対策
- 入力・出力処理設計
- HTML と SQL 組立て処理設計
- HTTPS

## 推測される典型問題例

公式サンプル非公開のため、徳丸本・体験記から想定される代表的なスタイルを示す。

### 例1：パスワード保管（知識＋設計判断）

> Webアプリケーションでパスワードをデータベースに保存する際の設計として、**最も適切**なものはどれか。
>
> a. パスワードを SHA-256 で1回ハッシュ化して保存する
> b. パスワードを AES-256 で暗号化して保存し、復号キーは設定ファイルに置く
> c. ユーザーごとに異なるランダムな salt を付けて bcrypt でハッシュ化し、ハッシュ値と salt を保存する
> d. パスワード文字列を Base64 エンコードして保存する

**正解：c**
ハッシュは不可逆、salt でレインボーテーブル無効化、bcrypt のコスト係数で総当たり対策まで備える。

### 例2：Cookie 属性（設計判断）

> セッション ID を格納する Cookie に**必ず設定すべき**属性の組み合わせとして適切なものはどれか。
>
> a. `Secure` ＋ `HttpOnly` ＋ `SameSite=Strict または Lax`
> b. `HttpOnly` ＋ `SameSite=None`
> c. `Secure` ＋ `Domain=.example.com` ＋ `Path=/`
> d. `Expires` の長期設定 ＋ `HttpOnly`

**正解：a**
XSS 経由の窃取（HttpOnly）・盗聴（Secure）・CSRF（SameSite）をまとめて防ぐ。

### 例3：コード提示型・脆弱性特定

> 以下のサーバーサイドコードに含まれる脆弱性として最も適切なものはどれか。
>
> ```python
> sql = "SELECT * FROM users WHERE name='" + request.GET['name'] + "'"
> cursor.execute(sql)
> ```
>
> a. クロスサイトスクリプティング
> b. SQL インジェクション
> c. オープンリダイレクト
> d. CSRF

**正解：b**
対策はプレースホルダ（パラメータ化クエリ）。ユーザー入力を SQL 文に直接連結している点が問題。

### 例4：パスワードリセット設計（実務判断）

> パスワードリセット機能の設計として**不適切**なものはどれか。
>
> a. リセット用 URL は推測困難なトークンを含め、15〜30分で失効させる
> b. リセット申請時、登録済み・未登録にかかわらず同一のメッセージを返す
> c. 「該当メールアドレスが登録されていません」と表示してユーザーに知らせる
> d. リセット完了後は他端末の既存セッションを無効化する

**正解：c**
ユーザー列挙攻撃の温床になる。応答メッセージは登録有無を区別しない。

### 例5：OAuth/OIDC（プロトコル理解）

> OpenID Connect の認可リクエストに含める `state` パラメータの目的として正しいものはどれか。
>
> a. アクセストークンの有効期限を指定する
> b. CSRF を防ぐため、リクエストとレスポンスの対応を検証する
> c. ユーザーの権限スコープを示す
> d. ID トークンの署名アルゴリズムを指定する

**正解：b**
state はセッションに保持し、IdP から戻ってきた値と照合して攻撃者開始フローを弾く。

### 例6：HTTPS / TLS 設定

> Web サーバーで有効化すべき TLS 設定として**最も適切**なものはどれか。
>
> a. TLS 1.0 / 1.1 / 1.2 / 1.3 をすべて有効化する
> b. TLS 1.2 と 1.3 のみを有効化し、ECDHE による鍵交換と AES-GCM を優先する
> c. SSL 3.0 を含め下位互換のため広く有効化する
> d. RSA 鍵交換を必須にして前方秘匿性を高める

**正解：b**
古いバージョンは POODLE・BEAST 等の脆弱性あり。ECDHE で前方秘匿性、AES-GCM で認証付き暗号。

### 例7：ファイルアップロード設計

> アップロードされたファイルが PHP として実行されることを防ぐ設計として**最も効果がある**ものはどれか。
>
> a. ファイル名から `.php` 拡張子を削除する
> b. Content-Type ヘッダーで画像系のみ許可する
> c. ファイルを Web 公開ディレクトリ外に保存し、ダウンロードはアプリ経由にする
> d. アップロード後にウイルススキャンを実行する

**正解：c**
拡張子・Content-Type は改ざん可能。物理的に Web サーバーが解釈できない場所に置くのが本質的対策。

## 学習戦略

1. **公式テキストを3周精読**する（合格者の共通ルート）
2. **徳丸本**で各脆弱性の発生原理と対策コードを実例ベースで理解する
3. **「攻撃 → なぜ成立するか → どう設計で防ぐか」のセット**で覚える（暗記NG）
4. 本プロジェクトのスライドで**全体俯瞰 → 徳丸本で深掘り → 推測問題で自己テスト**のループを回す
5. コード提示型問題に備えて、SQL／HTML出力／OS コマンド呼び出しの**「脆弱なコード」と「正しいコード」を対で**読めるようにする
